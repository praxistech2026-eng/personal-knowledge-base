# AI Center 变更流程

> 最小闭环：发现 → 定位 → 修复 → 验证 → 归档。
> 原则：实时状态写 Homepage / watchdog，长期事实写档案库；不能核实的原样保留。

## 1. 变更类型

| 类型 | 触发场景 | 必做动作 |
|------|----------|----------|
| 新增服务 | 新装一个软件、起一个新容器、接入一个新平台 | 建 README，补 Homepage，补巡检，补索引 |
| 升级服务 | 版本升级、容器重建、参数变更 | 先记当前状态，再改配置，再验收 |
| 改配置 | 端口、路径、环境变量、模型路由 | 先看有没有影响面，再改 docs + runtime |
| 换 key | API Key、Token、登录凭证更新 | 先补凭证总册，再改运行态，再验证 |
| 下线服务 | 彻底停用或迁移 | 把状态改成 `not-running` / `installed-not-running`，并留归档说明 |
| 归档服务 | 不再维护但需要保留历史 | 保留 README / 历史记录 / 变更说明，别删光 |

## 2. 标准流程

### A. 发现
- 用户提出需求、巡检发现异常、或者新服务上线。
- 先判断是不是“真变更”。
- 不能核实的先标记，不猜。

### B. 定位
- 查 `services/<name>/README.md`
- 查 `operations/巡检项总表.md`
- 查 `homepage/services.yaml` / `homepage/bookmarks.yaml`
- 查 `docs/README.md` / `docs/WIKI-索引.md`
- 查凭证总册是否涉及新 key

### C. 修复
- 实时状态先改 Homepage / watchdog。
- 长期事实再改档案库。
- 结构变化再改拓扑图。
- 只改需要改的地方，不扩散。

### D. 验证
- `docker ps`
- `ss -tlnp`
- `curl` / `python3 homepage-link-watchdog.py`
- `python3 ai-center-health-watchdog.py`
- 如涉及备份，再跑备份巡检

### E. 归档
- 更新对应服务 README
- 更新运维总表
- 更新 Homepage 配置
- 更新 WIKI 索引
- 必要时更新拓扑图和总览 README
- 写明日期、结论、未核实项

## 3. 新增服务接入流程

1. 先确认服务是否真的存在。
2. 建 `services/<name>/README.md`。
3. 记录基础信息：类型、端口、地址、运行方式、状态。
4. 补 `USAGE_MANUAL` / `RUNBOOK` 相关内容，至少把健康、备份、告警说清。
5. 加入 `homepage/services.yaml` 或 `homepage/bookmarks.yaml`。
6. 加入 `operations/巡检项总表.md`。
7. 更新 `docs/README.md` 和 `docs/WIKI-索引.md`。
8. 跑一轮 watchdog，确认没有假警。

## 4. 最小故障闭环

1. 发现：巡检或用户反馈。
2. 定位：先看端口、容器、进程、日志。
3. 修复：只改一个最可能点，别同时乱改多个变量。
4. 验证：复跑 watchdog / curl / 端口检查。
5. 归档：更新 README、Homepage、运维总表。

## 5. 写档案库 vs 写 Homepage

### 写档案库
- 永久在线事实
- 服务用途
- 架构关系
- 端口 / 路径 / 运行方式
- 凭证用途（不写明文密钥）
- 变更结论

### 只写 Homepage / 巡检
- 当前是否在线
- 临时 401 / 403
- 最近一次巡检结果
- 瞬时资源波动
- `installed-not-running`

## 7. 长期维护规则

- 实时状态写 Homepage / watchdog，不写成长期事实。
- 长期事实写档案库，不用“感觉在线”。
- 不能核实的内容保留原样。
- 新增服务必须同时更新：README、Homepage、巡检、索引。
- 重大变更必须留下归档记录，防止下次接手的人靠猜。
