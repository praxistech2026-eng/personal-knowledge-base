# AI Center 档案库总则

> 本文件是 AI Center 档案库的最高执行规范。凡是整理计划、方法论、巡检规则、模板、自动化脚本与专项文档，均以本文件为准；如有冲突，以本文件为准。
> 当前执行版本见 `[AI-Center-档案库V1](./AI-Center-档案库V1.md)`。

---

## 1. 总目标

AI Center 档案库不是文档堆，而是一套可长期运维的知识与资产系统。它的目标只有五个：

1. **应进尽进**：所有已知信息都尽量纳入档案库。
2. **可核验**：档案里写的内容必须能被事实、配置、日志、API、进程、容器或可追溯来源支撑。
3. **可维护**：文档不是一次性产物，必须能持续更新。
4. **可自动化**：能自动发现、自动归档、自动巡检、自动告警的，尽量自动化。
5. **可交接**：后续接手的人拿到的是完整上下文，不是零散结论。

档案库覆盖以下全部对象：

- 服务
- 工具
- 平台
- 基础设施
- 设备
- 账号
- 域名
- 证书
- 密钥
- 配置
- 接口
- 迁移痕迹
- 调研过程
- 设置思路
- 已解决问题
- 未解决问题
- 经验条目

只要是和 AI Center / 家庭 AI 中枢 / HTPC / 家庭网络 / 相关服务运维有关的事实，都属于归档范围。

## 3. 信息分层

档案库里的信息必须分层，不允许混写：

- **事实层**：实测得到的当前事实
- **现态层**：当前运行状态、入口、端口、镜像、路径
- **历史层**：迁移来源、旧端口、旧镜像、旧配置、过往决策
- **问题层**：已解决问题、未解决问题、待排查问题
- **经验层**：反复出现的坑、排障顺序、配置技巧、判断方法
- **规则层**：可重复执行的标准、模板、巡检、自动化策略
- **待核实层**：暂时无法证实的内容，保留原样
- **待补充层**：已确认缺失，但尚未补档

原则：
- 现态不能冒充历史
- 历史不能冒充现态
- 推断不能冒充事实
- 未核实内容不能被写成已确认

## 4. 核心原则

### 4.1 先核实，再修改

先看实机、进程、容器、端口、API、配置、日志，再动文档。浏览器界面只算辅助证据，不是事实源。

### 4.2 未核实信息保持原样

无法证明的内容不许猜、不许补、不许硬改。可以标记为待核实，但不能伪装成现态。

### 4.3 错误只改已证实部分

发现文档错误时，只修已经证实的部分；没证实的部分继续保留原样。

### 4.4 现态与历史分离

当前状态、迁移来源、废弃记录必须分开写。历史快照不要改写成当前事实。

### 4.5 结构化状态优先于视觉呈现

Homepage、截图、卡片样式、缓存状态都不是事实本身。若视觉层与结构化状态冲突，以结构化状态、live API、日志、配置为准。

### 4.6 问题必须记录

所有已解决和未解决的问题都要记录，至少写清：

- 现象
- 结论
- 未解点
- 后续动作

问题不是临时聊天记录，必须沉淀成可检索经验。

### 4.7 档案库覆盖全 IT 资产

档案库不只收服务本体，还收账号、域名、证书、密钥、设备、配置、接口、迁移痕迹与设置思路。

### 4.8 自动入库优先

默认优先级：

1. 文件 / 目录监听
2. 消息入口
3. 网页 / 浏览器剪藏

能自动化就不要重复手工输入。不能自动化的，至少进入待补清单。

### 4.9 Homepage 是仪表盘，不是事实源

Homepage 的职责是展示服务状态和风险等级，必要时辅助判断下一步动作；它不是事实源。事实源仍然是 live API、watchdog、日志、配置和进程。

### 4.10 删除必须先判断“替换”还是“废弃”

用户说删某个对象时，先判断：

- 它是被新对象替换了
- 还是彻底废弃了

默认保守处理：先保留 baseline，再等确认。误删迁移基线的代价高于误保留旧档案。

## 5. 增删改规则

### 5.1 新增

新增任何对象，最小动作是：

1. 建立对应目录和 README
2. 补索引
3. 补总览
4. 补凭证 / 来源信息
5. 补巡检 / 备份 / 告警规则
6. 写变更记录

### 5.2 修改

只要事实变化，就要同步相关文档：

- README
- WIKI 索引
- 拓扑图
- 总览
- 凭证总册
- 巡检 / 告警文档
- 变更记录

派生文档不能长期脱离事实源。

### 5.3 删除

删除是高风险动作，必须走保守流程：

1. 标记候选删除
2. 扫描所有引用面
3. 判断是否存在替换节点
4. 保留必要历史留痕
5. 再执行删改

如果证据不足，先标待定，不直接删。

## 7. 巡检规则

### 7.1 分层巡检

巡检任务按粒度分层：

- **分钟级**：核心可用性、关键入口、致命故障
- **小时级**：服务状态、链接可达、备份新鲜度
- **日级**：容量、慢性风险、漂移检查
- **周级**：结构审计、文档一致性、遗留问题清理

### 7.2 同层聚合

同一层级的巡检尽量放在同一时段执行，减少碎片化通知。

### 7.3 健康静默

无异常时可以不通知。不要用“没问题”刷屏。

### 7.4 异常只打相关项

ABC 三个巡检任务里，只有 B 异常，就只反馈 B；A 和 C 不要跟着发。

### 7.5 异常反馈必须可决策

反馈内容至少要足够回答：

- 发生了什么
- 影响什么
- 可能原因是什么
- 建议下一步做什么
- 是否需要你立刻处理

异常通知不是播报，是辅助决策。

---

## 8. 告警规则

微信告警统一使用 C 模板：

- 问题
- 影响
- 建议
- 是否需要立刻处理

如果一轮巡检没有异常，不发消息。只把真正需要动作的内容送出来。

---

## 9. 问题 → 经验 → 规则

这条链路必须打通：

1. 发现问题
2. 记录问题
3. 区分已解决 / 未解决
4. 提炼经验
5. 将反复出现的经验升级成规则或模板
6. 将规则嵌入巡检、自动化和文档

如果问题只停留在记录，不转化成经验和规则，就等于没有沉淀。

---

## 10. 执行优先级

执行时按以下顺序判断：

1. 实测证据
2. live API / watchdog
3. 配置文件
4. 进程 / 容器 / 端口
5. 日志
6. 文档
7. 口述 / 推断

冲突时，证据等级高的覆盖证据等级低的。

---

## 11. 最小化操作原则

用户只做必要输入，其余尽量自动化。

自动化优先级：

- 自动发现
- 自动建档
- 自动同步索引
- 自动同步总览
- 自动巡检
- 自动告警
- 自动生成待补清单

只有自动化做不到的部分，才回退到人工。

---

## 12. 版本与同步

任何实质变更都要：

- 落到文件
- 同步索引
- 同步总览
- 同步相关方法论
- 同步巡检规则
- 必要时同步 Git

档案库不是聊天内容，不能只停在对话里。

---

## 13. 冲突处理

如果某个专项文档、计划、模板、脚本和本总则冲突：

- 以本总则为准
- 再修专项文档
- 再修模板
- 再修自动化

总则是上层规则，不是附录。
