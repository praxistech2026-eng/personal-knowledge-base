# AI Center 对象首次归类表（v0.1）

> 目标：把当前散落的对象先按**主归类**落位，后续再做依赖矩阵和项目状态卡。
> 规则：**一个对象可以跨视图出现，但只能有一个主归类。**

## 0. 归类规则

1. **物理层**：回答“东西放在哪”。
2. **连通层**：回答“东西怎么连”。
3. **运行层**：回答“系统里实际跑什么”。
4. **能力层**：回答“系统能提供什么能力”。
5. **项目层**：回答“现在在做什么事”。
6. **项目状态层**：回答“下次捡起来时从哪里继续”。

### 额外规则

- **运行层**和**能力层**不要混：前者是服务实体，后者是可复用能力。
- **项目层**和**项目状态层**不要混：前者是事情本身，后者是项目续跑快照。
- **Homepage / Codex / Hermes / OpenClaw** 这类对象有跨层属性，必要时主归类放在“运行层”或“连通/控制面”，但必须在备注里写清楚。

### 运行层里的跨层对象说明

- **Hermes / OpenClaw**：主归类放运行层，但它们同时承担“入口/控制面”的角色。
- **Homepage**：主归类放运行层，但它主要承担状态展示和入口聚合，不应被当作事实源。
- **LiteLLM / Hindsight**：主归类放运行层；它们对应的能力分别是“模型路由”和“记忆/检索”。

### 项目层的判断标准

一个东西是否值得被放在项目层，主要看两点：

1. **它是不是一次明确的工作包**。
2. **它最后能不能沉淀成能力、规则或模板**。

能沉淀的，最后要升到能力层或规则层。

## 6. 项目状态层（L6）

项目状态层不是项目本身，而是项目的**续跑快照**。

| 状态对象 | 说明 | 是否必须 | 备注 |
|---|---|---|---|
| 当前阶段 | 发现 / 收敛 / 设计 / 实施 / 验证 / 维护 / 暂停 | 是 | 项目现在在什么阶段 |
| 已完成 | 已经落地的成果 | 是 | 只列关键完成项 |
| 下一步 | 下次回来先做什么 | 是 | 只保留 1~3 件 |
| 阻塞点 | 卡在哪 | 是 | 不能只写“待处理” |
| 最近决策 | 为什么这么选 | 是 | 方便重启时不重复争论 |
| 依赖项 | 依赖哪些服务/能力/人 | 是 | 防止断链 |
| 最后更新时间 | 最近一次状态刷新 | 是 | 用于判断是否过期 |
| 风险 | 当前最可能出问题的地方 | 建议 | 高价值项目建议加 |
| 需要你确认的事 | 需要用户拍板的点 | 建议 | 保持最小交互 |
| 接手入口 | 下次从哪个文件/章节开始看 | 建议 | 提高续跑效率 |

### 项目状态层的定位

- 它是**状态机快照**，不是备忘录。
- 它是**重启入口**，不是历史堆栈。
- 它必须和项目层配套存在。

## 8. 当前结论

### 先收口的顺序

1. **先把物理层和连通层定死**，避免网络和宿主机边界继续混。
2. **再把运行层和能力层分开**，避免“一个服务既是服务又是能力”造成混乱。
3. **最后给每个项目补项目状态层**，解决你说的“重新捡起来没有上下文”。

### 一句话结论

**AI Center 的正确结构不是“设备 + 服务 + 项目”的并列，而是“物理 / 连通 / 运行 / 能力 / 项目 / 项目状态”的分层系统。**
