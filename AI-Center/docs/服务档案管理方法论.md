# 服务档案管理方法论 — 归档标准 + 凭证 + 联动机制

> 本文档梳理"新增一个软件/服务"时，应该在 AI Center 留下哪些档案才算完整。
> 适用对象：所有 services / tools / platforms / infrastructure 新增项。
> 最高约束：`[AI-Center-档案库总则](./AI-Center-档案库总则.md)`；若与总则冲突，以总则为准。

## 一、结论先行：服务档案"归档标准 + 凭证"清单

新增任何软件/服务时，**必须**在 AI Center 留下以下 6 类产物（"归档标准 + 凭证记录"）：

| # | 产物 | 路径模板 | 必填 | 说明 |
|---|------|---------|------|------|
| ① | **服务 README** | `services/<name>/README.md` 或 `tools/<name>/README.md` 或 `platforms/<name>/README.md` 或 `infrastructure/<name>/README.md` | ✅ | 基础信息表（类型/端口/路径/PID/状态） + USAGE_MANUAL（健康检查/备份/告警固定字段） + 架构定位 + 常用命令 + 迭代记录 |
| ② | **调研/选型报告** | `services/<name>/调研报告-<name>.md`（如有必要） | 🟡 选填 | 仅复杂项目需要：横向对比 + Ubuntu 适配 + 风险分析 |
| ③ | **WIKI 索引条目** | `docs/WIKI-索引.md` 内的一行链接 | ✅ | 让人能从索引跳进服务 |
| ④ | **拓扑图更新** | `docs/AI-Center-拓扑图.md` 的 ASCII 图 + 服务详情 | ✅ | 反映"这个服务在系统里的位置" |
| ⑤ | **总览 README 更新** | `docs/README.md` 服务目录表 | ✅ | 让人能扫一眼看到所有服务 |
| ⑥ | **凭证备忘录** | `credentials/系统凭证备忘录.md` 对应段落 | ✅（如有凭证） | API Key / token / 密码，写明存哪、不写明文 |

**你之前说的"归档标准"实际上就是 ①②③④⑤** — 我帮你数了一下现有约定，刚好是这五份：
1. 服务 README
2. 调研报告
3. WIKI 索引
4. 拓扑图
5. 总览 README

**凭证是第六件**，但你说"还有个凭证记录" — 我确认了下，AI Center 已经有 `credentials/` 目录和 `系统凭证备忘录.md` 在跑，**只是历史上不是所有服务都补全了**（详见第三节漏洞清单）。

### ① 服务 README（最核心，缺这个就不算归档完成）

```markdown
# <服务名>

| 属性 | 值 |
|------|-----|
| **类型** | Docker 容器 / npm 全局 / pip / systemd / 手动二进制 |
| **镜像/包** | <image:tag> 或 <pip package> |
| **监听端口** | <端口> |
| **数据/配置目录** | <绝对路径> |
| **宿主机挂载** | <host:container> |
| **运行机器** | <IP/hostname> |
| **访问地址** | http://...:端口 |
| **安装日期** | YYYY-MM-DD |
| **运行状态** | 🟢 运行中 / 🟡 已安装未运行 / 🔴 故障 / ⚫ 外部 |

---

## 架构定位
（1 段 ASCII 图，标出它依赖谁、被谁依赖）

## 在本生态中的角色
（1 句话：这个服务解决什么问题）

## 常用命令
（启停/重置密码/查日志/升级/备份 5 类命令）

## USAGE_MANUAL

### 健康检查

| 字段 | 值 | 说明 |
|------|----|------|
| 检查对象 | <端口 / 进程 / URL / 容器 / Cron> | 具体检查谁 |
| 检查命令 | <curl / ss / docker ps / systemctl ...> | 实际执行的命令 |
| 判据 | <200 / 端口监听 / PID 存在 / healthy> | 什么算通过 |
| 频率 | <每次变更后 / 每 5 分钟 / 每小时> | 谁来检查 |
| 失败动作 | <重启 / 告警 / 人工介入> | 检查失败后怎么处理 |

### 备份

| 字段 | 值 | 说明 |
|------|----|------|
| 备份对象 | <数据目录 / 配置 / 凭证索引 / DB> | 备什么 |
| 备份方式 | <tar / rsync / pg_dump / snapshot> | 怎么备 |
| 频率 | <hourly / daily / before upgrade> | 多久备一次 |
| 保留策略 | <7 天 / 30 份 / 轮转> | 保留多久 |
| 恢复命令 | <还原命令 / 回滚步骤> | 出事怎么恢复 |

### 告警

| 字段 | 值 | 说明 |
|------|----|------|
| 告警条件 | <health fail / disk full / backup fail> | 什么情况要通知 |
| 通知渠道 | <微信 / 飞书 / 邮件 / 终端告警> | 发给谁 |
| 兜底动作 | <自动重启 / 暂停写入 / 人工处理> | 第一响应做什么 |
| 升级路径 | <自动化优先，失败后人工> | 谁接手 |

### 恢复 / 回滚

（可选，但复杂服务建议保留一段，写清恢复顺序、回滚条件、回滚后验证命令。）

## 凭证与配置

（如有：API Key/Token 存哪；不要写明文）

## 迭代记录
| 日期 | 操作 |
|------|------|
| YYYY-MM-DD | 初始部署 / 升级 / 改端口 / 加备份 |
```

### ② 调研报告（仅复杂项目）

什么时候需要：

- 跨多种方案选型（如 OpenList vs Cloudreve vs Nextcloud）
- **服务替换/迁移**（如 AList → OpenList）：调研对比 + 完整卸载 + 档案迁移 + 新服务部署，本次 OpenList 替换 AList 即典型案例
- 涉及多版本/多平台/多架构适配
- 有明显风险点（风控/合规/资源占用）

模板：见 `services/openlist/调研报告-OpenList.md`

按现有段落（API Keys / 服务地址 / 飞书 / 微信 / 数据库 / Docker 容器 / 服务器 SSH）分类加。

**关键纪律**：

- ✅ 写"完整值在 `~/.hermes/.env` 的 `ENV_VAR_NAME`"
- ❌ 不要在 markdown 里写真 key 明文（容易被同步/截图/搜索引擎抓取）

### 四、问题沉淀与经验库

每次遇到问题，不管最后是否解决，都要在档案库里留下可检索记录：

- **已解决问题**：记录现象、根因、修复动作、验证结果、复盘结论
- **未解决问题**：记录现象、当前判断、阻塞点、下一步排查方向
- **经验条目**：把反复出现的坑、误判、配置技巧、排障顺序沉淀成可复用经验

原则：
1. 只写事实，不写感觉。
2. 记录结果，也记录未完成的部分。
3. 让后来的人看到就能直接接手，而不是只能看结论。
4. 所有问题记录都要可检索、可回溯、可继续补充。

### 🟠 中度漏洞：凭证备忘录与拓扑图/服务 README 双写

**现象**：

- `credentials/系统凭证备忘录.md` 记了 Ark API Key 明文前缀 `dd94ff...7817`

**修复**：

- 凭证的"完整值"一律只写在 `~/.hermes/.env`（chmod 600）
- markdown 文档统一只写"环境变量名"和"前缀/后缀"（用于人眼识别，不构成泄露）
- 建立"凭证登记表"和"凭证存储"分离：**登记表** = 哪些服务有凭证、变量名是什么、轮转周期；**存储** = 实际值在 `.env` / 密钥管理服务

### 🟢 流程漏洞：缺一份"新增服务 Checklist"

**现象**：你现在只能凭记忆去补全 5+1 件套。

**修复**：本文档就是答案。每次新增服务，**打开本方法论 → 按 ①→⑥ 顺序执行 → 最后回到 WIKI/拓扑/README 链接一遍**。

## 四、推荐落地流程（写给未来的你）

### 新增一个服务时：

```
1. 决策：装在哪、解决什么问题、用什么方式（Docker/包/源码）
2. 执行：实际部署（先备份现有数据/配置）
3. 当场记录：把"基础信息表"先写进 services/<name>/README.md（哪怕后面再补全）
4. 写入凭证：API Key/Token 进 ~/.hermes/.env，并在 credentials/系统凭证备忘录.md 登记
5. 更新拓扑：ASCII 图 + 服务详情小节
6. 更新 WIKI：分类加一行
7. 更新总览 README：服务目录加一行
8. 接入监控（如有）：Homepage / 健康检查
9. 接入备份：把数据目录加入现有备份流水线
10. 写一行追加到 AI-Center-拓扑图.md 的"更新日志"表
```

### 删除/下线一个服务时：

```
1. 备份最后一次数据
2. 停止服务/容器
3. services/<name>/README.md 顶部状态改为"🔴 已下线 YYYY-MM-DD"
4. WIKI 索引加删除线 + 标注下线日期
5. 拓扑图 ASCII 灰化或删除对应方块
6. credentials 备忘录如有过期凭证，加"已撤销"标注
7. 更新日志追加
```

### 每次配置文件大改时（如 v3.25+ 改密码 hash 化）：

```
1. 在该服务 README 的"迭代记录"追加
2. credentials 备忘录：旧值标注"已轮转"、新值写新环境变量
3. 拓扑图：相关表/图更新
4. 不动 WIKI（除非类别变了）
```
