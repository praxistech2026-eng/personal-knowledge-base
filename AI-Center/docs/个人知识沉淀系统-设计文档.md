# 个人知识沉淀系统 — 设计文档

> 这份文档现在只回答一个问题：**你的知识库怎样才能稳定吃进真实世界的脏输入，并把它们编译成可持续使用的知识。**
>
> 结论先说：问题不在 Obsidian、Hermes、Codex 谁更先进，问题在 **解析层和归一化层没有被正式定义**。从 2026-06-11 开始，本系统以“四层架构 + Canonical Envelope + staging 编译”作为新的权威基线。

---

## 一、当前结论

### 1.1 本系统的真正目标

本系统不是“收文件的脚本集合”，也不是“聊天机器人顺手写笔记”。

本系统的目标是：

1. 把多源输入安全接住
2. 判断平台 / 类型 / 处理路线
3. 提取可回查的证据与内容
4. 统一成强约束内容包
5. 再编译成 staging 卡、正式笔记、LLM Wiki 页面或后续知识资产

### 1.2 这次改造纠正的核心误区

过去版本默认：

```text
输入 → 类型分流 → AI 摘要/分类 → 直接写笔记
```

这条链路太脆，原因有三个：

1. **解析层职责不清**：平台判断、canonical URL、失败原因、媒体证据没有被正式建模
2. **归一化缺失**：脚本里已经有 `canonical_url / capture_status / source_tags`，但没有成为统一标准
3. **知识编译和输入解析混在一起**：导致一旦抓取不完整，后面的分类/标签/笔记都跟着失真

新版本改为：

```text
输入层 → 解析层 → 归一化层 → 知识编译层
```

## 二、四层架构（正式版）

## 2.1 Layer 1 — 输入层（Input Layer）

职责：**只负责接住输入，不负责理解知识。**

当前已存在的入口：

- `~/PersonalKnowledge/_inbox/` 文件投递
- Mac vault → Git push → 服务器 pull → `_inbox/`
- 微信 / 分享文本 / 链接转发（部分已通）
- 本地音频/视频/文档文件
- URL / share text / pasted text

输入层必须保留：

- 原始文件路径
- 原始分享文本
- 原始 URL
- 输入时间
- 输入来源（Mac / 微信 / 本地文件 / 手工投递）

### 输入层铁律

- 不在这一层做知识判断
- 不在这一层做主题分类
- 不在这一层做最终摘要
- 失败也不能丢原料

---

## 2.2 Layer 2 — 解析层（Parsing Layer）

职责：**判断这是什么，以及下一步怎么处理。**

解析层要解决的不是“总结内容”，而是：

1. 这是 `video / audio / web / document / image / text` 哪一类？
2. 这是 `youtube / bilibili / douyin / xiaohongshu / generic_web / local_file` 哪个平台？
3. 该走哪条处理路径？
4. 有没有 canonical URL？
5. 有没有字幕、音轨、正文、描述、封面、关键词？
6. 如果失败，失败在哪一步？

### 当前已验证的解析层能力

脚本：`/home/shin/bin/process-inbox-safe.py`

已具备的真实字段和逻辑：

- `platform`
- `source_url`
- `canonical_url`
- `expanded_url`
- `video_id`
- `capture_status`
- `transcript_method`
- `source_tags`
- `platform_tag`
- `media_type`
- `capture_method_tag`
- `quality_tag`

这说明系统并不是没解析层，而是**解析层已经长出来了，但还没被正式定义成平台无关标准**。

### 解析层处理原则

- YouTube / B站：优先原生字幕与平台 metadata
- 抖音 / 小红书：优先 canonical URL + 页面 metadata + 浏览器兜底
- 本地音视频：优先转写与文件元信息
- 网页/文档：优先正文抽取与 canonical source
- 任何失败都必须产生“可回放、可复抓、可人工补录”的结果

---

## 2.3 Layer 3 — 归一化层（Normalization Layer）

职责：**把不同来源的脏输入统一成一份强约束内容包。**

这个统一内容包在本系统中叫：

> **Canonical Envelope**

它不是笔记，也不是最终知识卡。它是“知识编译之前的标准输入对象”。

Canonical Envelope 至少要统一这些维度：

- 输入类型
- 平台
- canonical URL
- 原始输入
- 可用正文/字幕/转写/描述
- 元数据
- 质量等级
- 失败原因
- 取证信息（何时抓、谁抓、用什么方法抓）

本层的详细字段定义见：

- `[[知识摄入解析层与统一内容包规范]]`

---

## 2.4 Layer 4 — 知识编译层（Knowledge Compilation Layer）

职责：**把统一内容包编译成知识资产。**

可能的输出包括：

- `Conclusions/_staging/*.md` 待确认卡
- 正式 Obsidian 笔记
- LLM Wiki summary / entity / concept / overview 页面
- 每日简报
- 标签、关系、专题索引

### 关键边界

知识编译层不应该反过来决定解析路线。

也就是说：

- 先有 canonical URL、正文、字幕、失败原因
- 再做摘要、标签、关系、分类

而不是反过来拿一段残缺内容硬做“知识笔记”。

---

## 三、Canonical Envelope（系统基线）

系统统一内容包的最小结构如下：

```yaml
id: uuid
source_type: video | audio | web | document | image | text
platform: youtube | bilibili | douyin | xiaohongshu | generic_web | local_file
input_kind: url | share_text | local_file | pasted_text
status: pending | active | manual_review | failed

source:
  raw_input: ""
  source_input_path: ""
  source_url: ""
  expanded_url: ""
  canonical_url: ""

identity:
  title: ""
  author: ""
  language: zh | en | unknown
  published_at: ""
  content_id: ""

artifacts:
  description: ""
  transcript: ""
  markdown_body: ""
  keyframes: []
  cover_image_url: ""
  attachments: []

metadata:
  duration_sec: 0
  duration_text: ""
  keywords: []
  hashtags: []
  source_tags: []

quality:
  capture_status: transcript_first | metadata_only | manual_review | failed
  capture_method: native_subtitle | asr | browser_metadata | extractor_failure | fallback
  transcript_method: ""
  completeness_score: 0.0
  fail_reason: ""

provenance:
  fetched_at: ""
  fetcher: process-inbox-safe.py
  extractor_version: v2
  content_hash: ""
```

说明：

- `capture_status` 是结果质量，不是处理器名字
- `capture_method` 是取数路径
- `transcript_method` 是更细粒度的技术实现
- `source_tags` 是确定性标签，不是最终知识标签

---

## 四、staging 卡不是原始层，是编译层产物

这一点必须写死。

`Conclusions/_staging/*.md` 的定位：

- 不是原始输入
- 不是唯一真相
- 不是最终知识
- 它是 **Canonical Envelope 向可读知识卡的第一层投影**

### 当前 staging 卡已存在的映射字段

以 `process-inbox-safe.py` 当前输出为例，已经有：

- `capture_status`
- `source_url`
- `canonical_url`
- `platform`
- `platform_tag`
- `media_type`
- `capture_method_tag`
- `quality_tag`
- `source_tags`
- `video_id`
- `author`
- `duration_sec`
- `duration_text`
- `transcript_method`
- `cover_image_url`

### 后续统一原则

staging frontmatter 只保留两类字段：

1. **检索/筛选必需字段**
2. **回查证据必需字段**

像“知识性解释”“高层洞察”属于正文，不属于原始 frontmatter。

## 五、四类核心输入的推荐链路

## 5.1 视频

### 主路径

```text
URL/share text
→ 识别平台
→ 提取 canonical URL / metadata
→ 优先字幕
→ 次选音轨转写
→ 再不行用浏览器 metadata 兜底
→ 进入 Canonical Envelope
→ 编译 staging 卡
```

### 平台策略

- **YouTube**：字幕优先；必要时补 JS challenge / cookies
- **B站**：metadata 可稳定；字幕/音轨按样本判断
- **抖音**：`yt-dlp` 不稳定时走 canonical + 浏览器 metadata
- **小红书**：优先保真，不要让 404 跳转污染 canonical URL

### 质量等级

- `transcript_first`：拿到字幕或高质量转写
- `metadata_only`：只有 metadata，没有正文级文本
- `manual_review`：抓到部分信息，但需要人工介入
- `failed`：只有原始输入与失败原因

## 5.2 音频

### 主路径

```text
本地文件 / 远程音频
→ 媒体识别
→ Whisper 转写
→ transcript 清理
→ Canonical Envelope
→ staging / 正式知识卡
```

### 原则

- 音频知识沉淀的主产物是 transcript，不是摘要
- 摘要是编译层动作，不是解析层动作

## 5.3 网页

### 主路径

```text
URL
→ canonical source 识别
→ 正文抽取（Jina/页面抽取/HTML fallback）
→ metadata + markdown_body
→ Canonical Envelope
→ staging / summary / wiki 页面
```

### 原则

- 优先 canonical source，不拿日志当真相
- 抓不到正文也要保留 title / url / fail_reason

## 5.4 文档

### 主路径

```text
PDF/Office/Markdown/Text
→ 格式识别
→ 文本抽取（PyMuPDF / python-docx / openpyxl / python-pptx / markitdown 等）
→ 结构保留
→ Canonical Envelope
→ 编译知识卡
```

### 原则

- 文档解析先保结构，再谈总结
- 表格、标题层级、链接、页码边界属于上游资产，不应在一开始就丢掉

## 六、标签与分类策略（新基线）

标签分两层，不再混写。

## 6.1 确定性标签（解析层产出）

这些标签必须由规则生成，不交给 LLM 拍脑袋：

- `platform: youtube`
- `media_type: video`
- `capture_status: metadata_only`
- `capture_method: browser_metadata`
- `input_kind: share_text`

当前系统里的这些字段：

- `platform_tag`
- `media_type`
- `capture_method_tag`
- `quality_tag`
- `source_tags`

都属于这一层。

## 6.2 语义标签（知识编译层产出）

这些标签在拿到 transcript / markdown_body / description 后再生成：

- 主题
- 人名 / 品牌 / 产品
- 行业词
- 项目/问题域
- 关系与归属

### 重要原则

- 先有确定性标签，后有语义标签
- 先有统一内容包，后有分类
- 分类是投影，不是原始事实

---

## 七、与 LLM Wiki 的关系

本系统不再把 LLM Wiki 当成“输入层工具”，而是把它放回正确位置：

> **LLM Wiki 是组织层 / 编译层，不是热路径摄入层。**

也就是说：

- 原始输入先进入 `_inbox/`
- 经过解析和归一化
- 生成 staging 卡或结构化内容包
- 再由后续编译流程更新 wiki / summary / concept pages

这和 Karpathy 的原始思路是一致的：

- raw sources
- wiki
- schema

你缺的不是 wiki，而是 raw → wiki 之间那段工程化中间层。

### 已有能力

- `_inbox/` 入口存在
- `process-inbox-safe.py` 已具备视频类 canonical / status / tags 雏形
- `Conclusions/_staging/` 已作为待确认层使用
- 视频链路已实测覆盖 YouTube / B站 / 抖音 / 小红书的不同降级状态
- failure-without-loss 原则已经在实际产物中体现

### 仍需补强

1. Canonical Envelope 还没有被脚本全局采用
2. 视频之外的网页 / 文档 / 音频也要对齐同一字段规范
3. staging frontmatter 仍带有“脚本临时字段”气味，尚未彻底统一
4. LLM Wiki / 主题页 / 正式入库的编译动作仍需继续规范化
