# AI Center 档案库重构 — 阶段执行提示词

> 目标：把 AI Center 从“资料索引”升级成“可接班、可巡检、可恢复、可持续迭代”的运维型档案库。
>
> 原则：
> 1. 先核实事实，再改文档。
> 2. 能自动化的尽量自动化，不能核实的保留原样。
> 3. 实时状态写 Homepage / 巡检，长期事实写档案库。
> 4. WIKI 索引保留为统一入口。
> 5. 拓扑图必须体现组织关系，不再只是服务堆叠。

---

## 阶段 0：现状体检与证据采集

**目标**：核对当前系统、文档、凭证、服务状态、巡检、Homepage、拓扑图，形成真实基线。

**输入范围**
- `docs/`
- `services/`
- `platforms/`
- `infrastructure/`
- `credentials/`
- 运行中的服务、端口、容器、巡检任务、Homepage 状态

**输出**
- 现状事实清单
- 问题清单
- 缺口清单
- 需要用户补充的信息清单

**验收标准**
- 所有“当前状态”都有证据
- 过时/冲突/遗漏全部被标记
- 不凭空猜测

**执行提示词**
```text
你现在只做【阶段0：现状体检与证据采集】。
目标：把 AI Center 当前的文档、凭证、服务状态、巡检、Homepage、拓扑图全部做一轮真实核验。
要求：
1. 先只读，不要修改任何文件。
2. 允许使用网络查询、官方文档、实机命令、文件读取。
3. 逐项核对 docs/、services/、platforms/、infrastructure/、credentials/、Homepage、巡检任务、容器状态、端口监听、关键 API 可达性。
4. 输出“事实清单 + 问题清单 + 缺口清单”，每一条都要标明证据来源。
5. 对无法核实的信息明确标记“未核实”，不要猜。
6. 对已经失效的内容标记“过时/需勘正”，但不要先改。
7. 最后给出下一阶段需要重构的范围建议。
验收标准：清单必须覆盖系统总览、拓扑、凭证、服务 README、巡检、Homepage、LLM 服务和关键依赖。
```

## 阶段 1：档案库结构规划

**目标**：把现有目录升级成长期可维护的运维结构。

**输出**
- 新目录结构草案
- 各目录职责说明
- 文档类型映射表
- 需要保留/合并/拆分的文件清单

**验收标准**
- 新结构能支撑新人接手
- 每类信息都有归属
- WIKI 索引保留

**执行提示词**
```text
你现在做【阶段1：档案库结构规划】。
目标：在保留现有内容的基础上，设计一套更适合长期运维的 AI Center 档案库结构。
要求：
1. 先分析当前目录结构的问题，再提出新结构，不要直接重写。
2. 保留 WIKI 索引作为总入口。
3. 目录要覆盖：总览、服务、平台、基础设施、凭证、运维、巡检、Homepage、历史快照。
4. 设计要能让一个完全不懂 AI 的运维工程师快速理解系统。
5. 说明哪些内容写入长期档案，哪些只留在 Homepage/巡检，不写进档案库。
6. 输出目录树草案 + 每类文档职责 + 迁移顺序。
7. 不要动手改文件，只出方案。
验收标准：结构必须支持后续补齐“服务档案标准、凭证明文、巡检、Homepage、变更流程”。
```

## 阶段 2：系统拓扑图与总览重画

**目标**：把现有拓扑图重画成能体现组织关系、依赖关系、入口关系、状态分层的系统地图。

**输出**
- 重新设计的拓扑图
- 服务关系矩阵
- 分层说明（入口层 / 门户层 / 服务层 / 模型层 / 数据层 / 备份层 / 通知层）
- `docs/README.md` 总览同步

**验收标准**
- 能一眼看出谁依赖谁
- 能一眼看出入口在哪
- 能区分实时状态与长期档案

**执行提示词**
```text
你现在做【阶段2：系统拓扑图与总览重画】。
目标：把 AI Center 拓扑图从“简陋的服务堆叠图”重画成“能体现组织关系、依赖关系、入口关系、状态分层”的系统地图。
要求：
1. 先阅读当前拓扑图和总览 README，找出不清晰、过时、误标、关系缺失之处。
2. 重新设计成“入口层 / 门户层 / 服务层 / LLM网关层 / 数据层 / 备份层 / 通知层 / 外部平台层”的分层结构。
3. 同时输出 Mermaid 图、ASCII 快图、服务关系表三种表达。
4. 明确区分：实时状态（运行中/未运行/外部）与长期档案（架构、凭证、变更、历史快照）。
5. 更新 `docs/README.md` 的总览目录，使其与拓扑图一致。
6. 保留 WIKI 索引，不要删除旧入口，必要时做重定向说明。
7. 不要先修别的文档，先把拓扑图和总览定稿。
验收标准：新人能通过拓扑图理解整个系统是怎么分层、怎么依赖、怎么访问、怎么维护的。
```

## 阶段 3：凭证体系全面重写

**目标**：把 `credentials/系统凭证备忘录.md` 扩成可接班的明文总册。

**输出**
- 全量明文凭证总册
- 缺失凭证待核实清单
- 明文警告头
- 按服务分类的凭证表

**验收标准**
- 所有能记录的凭证都记录
- 不能记录的明确写“待你提供”
- 有强警告：这是内网档案库，不许推 Git

**执行提示词**
```text
你现在做【阶段3：凭证体系全面重写】。
目标：把 `credentials/系统凭证备忘录.md` 扩成可接班的明文凭证总册。
要求：
1. 采用明文记录，并在文件头增加强警告：这是内网档案库，不许推 Git。
2. 覆盖所有可记录的凭证：IP、URL、端口、用户名、密码、API Key、token、数据库账号、平台账号。
3. 先整理出“已知凭证”和“缺失凭证”两张表，缺失项列出来等用户补。
4. 把 LLM 厂商、服务账号、数据库、系统登录、Webhook、通知通道全部分区写清楚。
5. 对已经失效的凭证要标“已失效/已替换/已弃用”，不要删掉历史痕迹。
6. 不要把长期事实写到别的文件里分散存放，凭证总册必须是唯一入口。
验收标准：一个新运维工程师拿到这份文件，能知道系统接入点、账号体系和关键登录信息在哪。
```

## 阶段 4：服务目录档案标准补齐

**目标**：每个服务都具备可维护的最小闭环。

**每个服务至少要有**
- `README.md`
- `USAGE_MANUAL.md`
- `搭建纪要.md`
- `RUNBOOK.md`
- `CHANGELOG.md`（必要时）

**验收标准**
- README 不只是状态表
- USAGE_MANUAL 包含健康检查 / 备份 / 告警 / 恢复
- RUNBOOK 按故障类型列排障步骤
- 搭建纪要解释“为什么这样搭”
- 变更历史留痕

**执行提示词**
```text
你现在做【阶段4：服务目录档案标准补齐】。
目标：按服务逐个补齐可维护的档案，形成可接班的服务目录。
要求：
1. 先按优先级处理核心服务，再处理平台和工具。
2. 每个服务至少补齐：README、USAGE_MANUAL、搭建纪要、RUNBOOK、CHANGELOG（如适用）。
3. README 重点写当前状态、入口、依赖、数据目录、访问方式。
4. USAGE_MANUAL 固定包含：健康检查、备份、告警、恢复/回滚。
5. RUNBOOK 按“起不来、访问失败、登录失败、数据异常、容器重启、限流”等常见故障分类。
6. 搭建纪要写清楚选型原因、替代方案、长期维护方向。
7. 遇到无法核实的信息，保留原样或标记未核实，不要硬改。
8. 每补完一个服务，就同步更新 WIKI、拓扑图、总览 README 和凭证总册。
验收标准：每个服务都能单独被一个新人维护，不依赖你的口头解释。
```

## 阶段 5：巡检体系整合 + Homepage 门户化

**目标**：把现有巡检整合成一套全面机制，并让 Homepage 同时承担导航与健康展示。

**输出**
- 统一巡检总入口
- 分级告警策略
- Homepage 状态展示
- 实时状态与长期事实边界说明

**验收标准**
- 打开 Homepage 能知道系统整体是否健康
- 巡检先全量采集，再按优先级简化
- 不把高频波动写进档案库
- 告警足够判断是否介入

**执行提示词**
```text
你现在做【阶段5：巡检体系整合 + Homepage 门户化】。
目标：把现有巡检整合成一套全面机制，并让 Homepage 既能导航又能展示健康状态。
要求：
1. 先梳理现有巡检项，找出重复、缺失、误报、噪音。
2. 新增主机资源巡检：CPU、内存、磁盘、inode、Docker 状态、关键进程。
3. 新增链路巡检：模型网关、备份链路、微信通知、Homepage 状态、数据库可达性、网络可达性。
4. 巡检输出要足够让我判断“需不需要介入”，但不要把高频波动写进档案库。
5. Homepage 既要能跳转，也要能展示服务是否可用、模型是否正常、链路是否在跑。
6. 区分实时状态与长期事实：实时状态交给 Homepage/巡检，长期事实写进文档。
7. 最后给出统一巡检机制设计图和告警分级标准。
验收标准：打开 Homepage 就能知道系统整体是不是健康，出了问题先看哪里。
```

## 阶段 6：变更流程、标准与最终收口

**目标**：把这套系统固化成长期可执行流程。

**输出**
- 变更管理流程
- 新增服务接入流程
- 服务下线流程
- 凭证更新流程
- 档案库维护标准
- 最终验收报告

**验收标准**
- 新增服务知道怎么进库
- 改 key 知道怎么更新
- 服务下线知道怎么归档
- 不再需要口头反复解释

**执行提示词**
```text
你现在做【阶段6：变更流程、标准与最终收口】。
目标：把 AI Center 档案库和运维体系固化成一套可长期执行的流程。
要求：
1. 基于前面阶段的结果，写清楚新增、升级、改配置、换 key、下线、归档的流程。
2. 把“哪些写档案库、哪些只写 Homepage/巡检”明确分开。
3. 为未来新增服务写一份标准接入流程，要求可复制执行。
4. 为未来故障处理写一份最小闭环流程：发现 → 定位 → 修复 → 验证 → 归档。
5. 汇总本次重构的所有改动文件和长期维护规则。
6. 不要再新增不必要的目录或分散文档，收口要简洁。
验收标准：以后新增一个服务，不需要我重新口头解释怎么归档。
```

## 推荐执行顺序

1. 阶段 0：现状体检
2. 阶段 1：结构规划
3. 阶段 2：拓扑图与总览重画
4. 阶段 3：凭证体系重写
5. 阶段 4：服务档案标准补齐
6. 阶段 5：巡检整合 + Homepage 门户化
7. 阶段 6：变更流程收口

---

## 关键约束

- 实时状态写 Homepage / 巡检，不写成长期事实
- 长期事实写档案库，不用“感觉在线”
- 凭证明文允许，但必须加强警告，禁止推 Git
- 拓扑图必须体现组织关系，不再是服务盒子堆叠
- 不能核实的内容保留原样
