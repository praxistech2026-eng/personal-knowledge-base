# AI Center 架构骨架（草案）

> 目标：把现在散在物理层、连通层、运行层、能力层、项目层里的东西拆开，避免一张图塞所有内容。
> 核心新增层：**项目状态层 / 项目重启层**。

## 1. 这份草案解决什么

你现在的问题不是“没有架构”，而是：

- 同一张图里混了设备、网络、服务、能力、项目
- 项目不是连续推进的，下次捡起来时缺上下文
- 你既需要长期稳定的中台能力，也需要每个项目的推进状态
- 文档和记忆都在用，但没有明确分工

所以架构必须拆成两条主线：

1. **稳定能力线**：可复用、可沉淀、长期存在
2. **项目状态线**：可重启、可续跑、可接手

## 3. 三个最重要的边界

### 3.1 依赖方向只能向下

```text
项目层 -> 能力层 -> 运行层 -> 连通层 -> 物理层
```

不能反过来。

### 3.2 中台只收能力，不收项目状态

像这些应该进入能力层：

- LiteLLM = 模型路由中台
- Hindsight = 记忆/检索中台
- 通知链路 = 告警中台
- OpenClaw / Hermes = 入口与执行控制面

像这些不应该被中台化：

- 某个单次迁移任务
- 某个服务的临时改造
- 某次模型上线/下线
- 某个网络拓扑调整

这些属于项目，不属于能力。

### 3.3 项目状态不能只放在记忆里

记忆只能保存偏好、规则、长期共识。

项目状态必须写进文档，因为它是可重启状态机，不是稳定事实。

## 6. 项目状态卡模板

```markdown
# 项目状态卡：<项目名>

## 1. 项目目标
- 这个项目要解决什么
- 为什么值得做

## 2. 当前阶段
- 发现 / 收敛 / 设计 / 实施 / 验证 / 维护 / 暂停

## 3. 当前结论
- 目前已确认了什么
- 哪些是事实，哪些只是推断

## 4. 已完成
- 列点

## 5. 下一步
- 下一步只写 1~3 件最关键的事

## 6. 阻塞点
- 缺什么信息
- 卡在哪个依赖
- 谁需要确认

## 7. 最近决策
- 为什么这么选
- 备选方案是什么

## 8. 依赖项
- 依赖哪些能力
- 依赖哪些服务
- 依赖哪些外部入口

## 9. 最近更新时间
- YYYY-MM-DD HH:mm
```

## 7. 项目重启摘要模板

这个是下次重新捡起项目时先看的东西，要求极短。

```markdown
# Restart Summary

- 项目名：
- 当前阶段：
- 已完成：
- 下一步：
- 阻塞点：
- 最近决策：
- 依赖项：
- 本次会话只要做什么：
```

### 规则

- 这页只写“接下来怎么继续”
- 不写长篇解释
- 不写历史大段复盘
- 不写和本次继续无关的内容

## 8. 文档和记忆怎么分工

### 文档负责
- 进度
- todo
- checkpoint
- 决策记录
- 变更记录
- 接手摘要

### 记忆负责
- 用户长期偏好
- 稳定规则
- 重复坑位
- 不太会变的共识

### 结论

**todo 和进度不能只存在记忆里。**

如果只靠记忆：
- 会漂
- 会丢上下文
- 会难以审计
- 会无法接手

如果只靠文档：
- 下次重新打开仍然找不到重点
- 缺少“该先看什么”的优先级

所以必须两者配合：

- 文档保存状态机
- 记忆保存稳定约束

## 9. 推荐的项目生命周期

```text
发现 -> 收敛 -> 设计 -> 实施 -> 验证 -> 维护 -> 暂停 / 废弃
```

### 每个阶段要回答什么

- **发现**：这事值不值得做？
- **收敛**：目标到底是什么？边界在哪里？
- **设计**：怎么拆层？怎么复用？
- **实施**：先做最小闭环还是一次铺满？
- **验证**：怎么证明真的可用？
- **维护**：怎么续跑？怎么升级？
- **暂停 / 废弃**：为什么停？历史怎么留？
