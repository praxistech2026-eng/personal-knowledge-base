# AI Center 巡检总控

> 这是 AI Center 统一巡检的总控层。
> 原则：**一个总控，多层探针；一个告警出口，多条子巡检。**
> 任何专题巡检都必须归属到总控的某一层，不允许漂成独立小系统。

## 1. 总控目标

把现在分散的巡检收口成一套可执行、可审计、可退场的机制：

- 统一巡检入口
- 统一状态命名
- 统一告警格式
- 统一归档位置
- 统一任务生命周期
- 统一“为什么存在”的说明

## 2. 现有巡检资产盘点

当前已经存在的巡检，不是很多套不同体系，而是同一套总控下的不同层：

### 2.1 资源域

现有资产：

### 2.2 服务域

- `operations/巡检项总表.md` 中的“核心服务巡检”

### 2.3 门户域

负责 Homepage、远程入口、状态展示与真实可达性一致性。

现有资产：
- `homepage-link-watchdog.py`
- `homepage/README.md`

### 2.4 数据与记忆域

### 2.5 档案库与配置审计域

## 3. 总控结构

### 3.1 统一层级

```
AI Center 巡检总控
├── 资源域巡检
├── 服务域巡检
├── 门户域巡检
├── 数据/记忆域巡检
├── 档案库/配置审计巡检
└── 告警/归档/验证
```

### 3.2 横切规则

所有巡检都必须遵守：

1. **先实测，再判断**
2. **只报异常，不刷屏**
3. **异常必须可行动**
4. **未核实内容保持原样**
5. **高频波动不进长期事实**
6. **巡检结果要回写到正确层**：实时状态进 Homepage / watchdog，长期规则进档案库

## 4. 统一任务卡模板

以后任何巡检任务都必须补齐这 8 项：

| 字段 | 说明 |
|---|---|
| 任务名 | 这项巡检叫什么 |
| 归属域 | 资源 / 服务 / 门户 / 数据 / 档案库 / 审计 |
| 为什么存在 | 它是为了解决什么问题 |
| 输入 | 读哪些文件、API、进程、容器、日志 |
| 探针 | 用什么命令 / API / watchdog |
| 判定 | 什么算正常，什么算异常 |
| 输出 | 写哪里、报哪里、谁来接 |
| 退场条件 | 什么情况下可以合并 / 废弃 / 归档 |

### 4.1 任务卡最低要求

如果一项巡检说不清下面三件事，就先别建：

- 它属于哪个域
- 它到底在防什么问题
- 它异常后要做什么

## 5. 现有巡检如何收口

### 5.1 现在保留的主巡检

当前应保留的主巡检只有这些：

- **资源巡检**：主机、容器、端口、磁盘、内存
- **服务巡检**：核心服务健康与在线性
- **门户巡检**：Homepage 与远程入口一致性
- **数据/记忆巡检**：备份、入库、retention、剪枝
- **档案库巡检**：文档一致性、模板、索引、拓扑、凭证、周审
- **配置审计巡检**：规则漂移、pending、变更审计

### 5.2 应该降级成“子项”的内容

以下内容不要再独立长成专题巡检：

- 单个服务的零散健康检查脚本
- 某次事故的临时检查清单
- 只服务一次迁移的临时核验项
- 没有退场条件的“永远保留”任务

这些都应回到总控下的某个域，或者升级成经验/规则，不要单独挂着。

## 6. 任务生命周期

每个巡检任务都必须经历这 5 个状态：

1. **创建**：说明为什么存在
2. **运行**：按固定频率执行
3. **验证**：确认输出真的有效
4. **收敛**：判断是否可以合并到别的任务
5. **退场**：废弃、归档或升级为规则

没有退场条件的任务，默认视为设计缺失。

## 7. 告警与归档

### 7.1 告警

统一走可行动格式：
- 问题
- 影响
- 建议
- 是否需要你立刻处理

### 7.2 归档

巡检产出只写三类东西：
- 现态：进 Homepage / watchdog
- 规则：进 `docs/AI-Center-档案库总则.md` / `docs/AI-Center-档案库V1.md`
- 经验：进问题记录 / 经验模板 / 变更记录

### 7.3 静默原则

无异常时静默，不发“没问题”消息。

### 7.4 session 保留策略

- **永久保留 raw**：重大变更、事故/故障复盘、恢复操作、配置/权限/凭证相关变更（不含明文 secret）、明确的用户纠错/偏好、首次出现的新模式、能反复引用的证据链。
- **只留摘要/元数据**：例行巡检、cron/heartbeat、重复性工具调用、低信息量聊天、纯状态查看、自动化噪音。至少保留 `session_id`、时间、来源、结论摘要、标签。
- **可以 prune**：已进入冷备且有摘要/元数据的低价值旧 raw，重复记录，空转会话，失败产物，明显的自动化噪音。
- **剪枝前置条件**：先冷备，后摘要，再 prune；没有冷备或摘要的 raw 不删。

## 10. 结论

现在不是再加新巡检，而是：

1. 把已有巡检挂回总控
2. 给每个巡检补任务卡
3. 合并重复项
4. 给每个巡检加退场条件
5. 让告警、归档、验证都回到同一套口径

**一句话：巡检不是很多套，巡检只能有一套总控。**
