# AI Center 档案库 V1（实施版）

> 这是 AI Center 档案库的 V1 执行基线。它负责把总则变成日常操作规则、模板、任务、边界和流程。
> 最高规范见 `[AI-Center-档案库总则](./AI-Center-档案库总则.md)`；若冲突，以总则为准。
> 巡检总控见 `[AI-Center-巡检总控](./AI-Center-巡检总控.md)`；V1 负责落地执行，不负责另起巡检体系。

---

## 1. V1 要解决什么

V1 的目标不是“把文档写多”，而是把档案库变成一个可长期运维的系统：

- 有明确边界
- 有固定模板
- 有统一巡检
- 有可行动告警
- 有问题沉淀
- 有经验回流
- 有自动化入口
- 有迭代机制

只要满足这 8 件事，V1 就算成立。

---

## 2. V1 的硬原则

### 2.1 事实优先

- 先看实机、配置、进程、容器、端口、API、日志
- 浏览器界面只作辅助证据
- 推断不能写成事实
- 未核实信息保持原样

### 2.2 记录优先

- 已解决问题要记
- 未解决问题也要记
- 经验必须沉淀
- 变更必须留痕

### 2.3 自动化优先

- 能自动发现就不手工抄
- 能自动归档就不重复写
- 能自动巡检就不人工盯
- 能自动告警就不靠人发现

### 2.4 删除保守

- 删除前先确认是不是“替换”而不是“废弃”
- 没有证据就先标待定
- 先保留 baseline，再做删改

### 2.5 派生文档服从事实源

- README、WIKI、拓扑、Homepage、凭证、巡检必须互相一致
- 结构化状态优先于视觉展示
- 如果冲突，以 live 证据为准

## 3. V1 范围

### 3.1 纳入范围

以下都属于 V1 归档范围：

- 服务
- 工具
- 平台
- 基础设施
- 设备
- 账号
- 域名
- 证书
- 密钥
- 配置
- 接口
- 迁移痕迹
- 调研过程
- 设置思路
- 已解决问题
- 未解决问题
- 经验条目

### 3.2 不纳入“现态层”的内容

以下内容可以进档案库，但不能直接当成当前事实：

- 用户口述但未核实的信息
- 老截图
- 历史快照
- 迁移来源
- 猜测
- 过时状态

它们只能进入历史层、待核实层、待补充层。

## 4. V1 标准产物

每个服务 / 设备 / 平台 / 关键对象，V1 至少应形成以下产物：

1. `README.md`
2. `USAGE_MANUAL.md` 或等价章节
3. `runbook.md` / `SOP.md`

6. `issues.md` 或问题记录
7. `experience.md` 或经验条目
8. `docs/WIKI-索引.md` 条目
9. `docs/AI-Center-拓扑图.md` 更新

## 6. 任务分层

### 6.1 分钟级任务

用于核心可用性和致命故障：

- 关键入口是否可达
- 核心服务是否存活
- 必须在线的容器/进程是否在跑

### 6.2 小时级任务

用于状态一致性和常规健康：

- 服务状态巡检
- 链接可达性
- 备份新鲜度
- 门户与文档是否漂移

### 6.3 日级任务

用于慢性风险：

- 容量变化
- 配置漂移
- 老问题是否复发
- 待补清单是否增长

### 6.4 周级任务

用于结构审计：

- README / WIKI / 拓扑 / Homepage 一致性
- 问题与经验是否回流
- 规则与模板是否需要升级
- 历史态与现态是否混写

### 6.5 调度原则

- 同层任务尽量同一时段执行
- 异常只通知相关项
- 无异常可静默
- 一条通知要能支持下一步判断

## 7. 巡检与告警边界

### 7.1 巡检边界

巡检只负责发现和判定，不负责替用户做所有决策。

### 7.2 告警边界

告警只发对人有用的信息，不发废话。

### 7.3 反馈边界

如果一轮巡检有 3 个任务，只有 B 异常，就只反馈 B。

### 7.4 决策边界

反馈必须至少回答：

- 出了什么问题
- 影响什么
- 为什么可能出问题
- 现在该怎么处理
- 要不要立刻处理

---

## 8. 增删改流程

### 8.1 新增流程

1. 识别对象
2. 核实事实
3. 建立目录 / README
4. 加索引
5. 加总览
6. 加凭证 / 来源
7. 加巡检 / 备份 / 告警
8. 写变更记录
9. 如有问题，立即补问题记录和经验条目

### 8.2 修改流程

1. 先确认事实变化
2. 只改已证实部分
3. 同步派生文档
4. 同步变更记录
5. 如触发经验复用，提炼成规则或模板

### 8.3 删除流程

1. 标记候选删除
2. 扫描引用面
3. 判断替换 / 废弃
4. 保留历史留痕
5. 执行删改
6. 反向检查残留引用

证据不足时，不删除，先标待定。

---

## 9. 自动入库流程

默认自动入库顺序：

1. 文件 / 目录监听
2. 消息入口
3. 网页 / 浏览器剪藏

自动入库触发后，优先做这几件事：

- 生成待补项
- 生成问题记录
- 生成经验候选
- 同步索引 / 总览

---

## 10. 问题沉淀流程

每次遇到问题，必须判断它属于哪一类：

- 已解决
- 未解决
- 反复出现
- 需要抽象成规则

处理顺序：

1. 记现象
2. 记结论
3. 记未解点
4. 记下一步动作
5. 若反复出现，提炼为经验
6. 若经验稳定，升级为规则或模板

---

## 11. V1 边界

V1 不做这些事：

- 不把未核实内容写成现态事实
- 不自动删除高风险对象
- 不让 Homepage 充当事实源
- 不让一个通知替代事实核验
- 不把经验直接当规则，必须经过重复验证

V1 只保证：

- 规则能执行
- 模板能复用
- 任务能巡检
- 异常能通知
- 问题能沉淀
- 经验能升级

---

## 12. V1 迭代机制

V1 不是终版，迭代规则如下：

- 每次出现重复问题，先补问题记录
- 连续出现 2 次以上，提炼经验
- 连续出现 3 次以上，考虑固化成规则或模板
- 因事实变化而变化的内容，优先改事实源
- 因流程变化而变化的内容，更新总则 / V1 / 方法论

迭代原则：

- 小步快改
- 先补缺口
- 先落文件
- 再补自动化
- 不追求一次写完，追求持续正确

---

## 13. V1 的判断标准

如果下面这些都能成立，V1 就算能用了：

- 新对象有固定落点
- 老对象能被巡检
- 异常能被模板化表达
- 问题能被记录
- 经验能被回流
- 规则能被升级
- 文档能和事实对齐

---

## 14. 和总则的关系

- 总则：最高规范
- V1：当前执行版本
- 方法论：服务级模板与归档细则
- 整理计划：阶段路线
- 巡检 / 告警：执行规则

如果冲突，以总则为准；如果总则没有写清，以 V1 为执行参考；如果 V1 也没有写清，再补总则或方法论。
