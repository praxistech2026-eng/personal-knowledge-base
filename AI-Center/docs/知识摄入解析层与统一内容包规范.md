# 知识摄入解析层与统一内容包规范

> 这份文档只解决一件事：**在进入知识编译层之前，系统应该把所有输入统一成什么样。**
>
> 不再允许“脚本各写各的 frontmatter，文档各讲各的字段”。

---

## 一、适用范围

本规范适用于以下输入：

- URL
- 分享文本（share text）
- 本地文件
- 粘贴文本
- 音频
- 视频
- 网页
- 文档
- 图片

目标不是覆盖一切细节，而是先统一最小公共结构。

---

## 二、解析层职责

解析层只做三件事：

1. **识别**：平台、类型、输入来源
2. **提取**：标题、作者、canonical URL、正文/字幕/转写、元数据
3. **判级**：这次抓取质量如何，下一步是否需要人工介入

解析层不做：

- 最终知识判断
- 主题总结
- evergreen note 编写
- 高阶语义分类

---

## 三、Canonical Envelope（正式结构）

## 3.1 顶层结构

```yaml
id: uuid
source_type: video | audio | web | document | image | text
platform: youtube | bilibili | douyin | xiaohongshu | generic_web | local_file
input_kind: url | share_text | local_file | pasted_text
status: pending | active | manual_review | failed
source: {}
identity: {}
artifacts: {}
metadata: {}
quality: {}
provenance: {}
```

---

## 3.2 source

```yaml
source:
  raw_input: ""
  source_input_path: ""
  source_url: ""
  expanded_url: ""
  canonical_url: ""
```

### 字段说明

| 字段 | 含义 | 必填 |
|---|---|---|
| `raw_input` | 原始分享文本 / 原始粘贴文本 | 建议 |
| `source_input_path` | 原始文件路径 | 文件输入时必填 |
| `source_url` | 用户最初提供的 URL | URL/分享输入时必填 |
| `expanded_url` | 短链展开后的 URL | 有则填 |
| `canonical_url` | 归一化后的标准链接 | 强烈建议 |

### 规则

- `source_url` 保留用户看到的原始入口
- `expanded_url` 用于保留短链展开后的真实链接
- `canonical_url` 用于做去重、归档、引用、后续重抓

### 不能犯的错

- 把 404 跳转页当 `canonical_url`
- 用日志截断链接充当原始链接
- 用转码后的临时地址覆盖用户原始入口

---

## 3.3 identity

```yaml
identity:
  title: ""
  author: ""
  language: zh | en | unknown
  published_at: ""
  content_id: ""
```

### 规则

- `title` 可以为空，但不能瞎编
- `author` 抓不到就留空
- `content_id` 优先用平台稳定 ID，例如 `video_id`
- `language` 不确定就写 `unknown`

---

## 3.4 artifacts

```yaml
artifacts:
  description: ""
  transcript: ""
  markdown_body: ""
  keyframes: []
  cover_image_url: ""
  attachments: []
```

### 定义

| 字段 | 含义 |
|---|---|
| `description` | 平台描述 / 简介 / 原始摘要片段 |
| `transcript` | 字幕或 ASR 转写全文 |
| `markdown_body` | 网页/文档抽取后的正文 Markdown |
| `keyframes` | 关键帧列表（后续扩展） |
| `cover_image_url` | 封面链接 |
| `attachments` | 关联附件路径或链接 |

### 原则

- 视频知识沉淀优先 `transcript`
- 网页/文档知识沉淀优先 `markdown_body`
- `description` 不能替代 transcript/body，只能做补充

---

## 3.5 metadata

```yaml
metadata:
  duration_sec: 0
  duration_text: ""
  keywords: []
  hashtags: []
  source_tags: []
```

### 规则

- `source_tags` 是**确定性标签**，不是语义标签
- `keywords/hashtags` 保留平台原生或抽取值，不要强行润色
- `duration_text` 是展示友好字段，`duration_sec` 是计算字段

---

## 3.6 quality

```yaml
quality:
  capture_status: transcript_first | metadata_only | manual_review | failed
  capture_method: native_subtitle | asr | browser_metadata | extractor_failure | fallback
  transcript_method: ""
  completeness_score: 0.0
  fail_reason: ""
```

### 字段边界

| 字段 | 作用 |
|---|---|
| `capture_status` | 结果质量等级 |
| `capture_method` | 这次主要靠什么路径拿到内容 |
| `transcript_method` | 更细的技术实现，例如 `yt-dlp-subtitles:zh` / `faster-whisper` |
| `completeness_score` | 可选，后续加入 |
| `fail_reason` | 失败或降级原因 |

### capture_status 定义

| 值 | 含义 |
|---|---|
| `transcript_first` | 拿到了可用字幕或高质量转写 |
| `metadata_only` | 只有标题/作者/描述/时长等 metadata |
| `manual_review` | 抓到了部分内容，但人工判断仍必要 |
| `failed` | 只有原始输入与失败原因 |

### capture_method 定义

| 值 | 含义 |
|---|---|
| `native_subtitle` | 平台原生字幕 / 自动字幕 |
| `asr` | 音轨转写 |
| `browser_metadata` | 靠浏览器页面元数据兜底 |
| `extractor_failure` | 提取器失败，只保留兜底结果 |
| `fallback` | 其他非主路径降级 |

---

## 3.7 provenance

```yaml
provenance:
  fetched_at: ""
  fetcher: process-inbox-safe.py
  extractor_version: v2
  content_hash: ""
```

### 规则

- `fetched_at` 必须是本次抓取时间
- `fetcher` 标明脚本或处理器
- `content_hash` 用于后续去重与幂等

---

## 四、平台识别规则

## 4.1 platform 的最小枚举

```text
youtube
bilibili
douyin
xiaohongshu
generic_web
local_file
```

后续可以扩展，但不要乱起名。

### 规则

- 平台名统一小写
- 平台字段只写平台，不写状态
- `local_file` 是平台，不是 `unknown`

---

## 五、frontmatter 映射规则

staging 卡的 frontmatter 不是 Canonical Envelope 原封不动复制，而是它的**可读映射层**。

## 5.1 保留字段（当前推荐）

```yaml
title:
created:
type:
status:
capture_status:
source_input:
source_url:
canonical_url:
platform:
platform_tag:
media_type:
capture_method_tag:
quality_tag:
source_tags:
author:
duration_sec:
duration_text:
transcript_method:
cover_image_url:
```

### 说明

- `platform_tag`、`media_type`、`capture_method_tag`、`quality_tag` 属于筛选友好字段
- `source_tags` 方便 Dataview / 批量筛选
- `canonical_url` 是后续重抓与去重关键字段

## 5.2 字段映射

| Frontmatter 字段 | Canonical Envelope 来源 |
|---|---|
| `source_input` | `source.source_input_path` |
| `source_url` | `source.source_url` |
| `canonical_url` | `source.canonical_url` |
| `platform` | `platform` |
| `platform_tag` | `platform` |
| `media_type` | `source_type` |
| `capture_status` | `quality.capture_status` |
| `capture_method_tag` | `quality.capture_method` |
| `quality_tag` | `quality.capture_status` |
| `transcript_method` | `quality.transcript_method` |
| `source_tags` | `metadata.source_tags` |

---

## 六、标签分层规范

## 6.1 确定性标签（解析层）

示例：

- `video`
- `youtube`
- `metadata_only`
- `browser_metadata`

这些由规则生成，不交给 LLM。

## 6.2 语义标签（编译层）

示例：

- AI agent
- 电商运营
- 小红书种草
- Prompt engineering

这些基于 transcript / body / description 生成。

### 铁律

- 没 transcript/body 时，不要硬打高置信主题标签
- 解析层标签和语义层标签不要混写成同一个字段来源

---

## 七、失败兜底规范

任何失败都不能静默丢失。

最小保留集：

```yaml
source_url:
canonical_url:
platform:
raw_input:
capture_status: failed | manual_review
fail_reason:
```

### 最低要求

- 能回查原始输入
- 能知道失败发生在哪
- 能知道下次从哪重试

### 绝对禁止

- 抓取失败后什么都不写
- 只报“失败”，不落卡
- 用空摘要假装成功

---

## 八、当前脚本与规范的关系

当前脚本：`/home/shin/bin/process-inbox-safe.py`

### 8.1 已对齐的部分

- 视频主卡已经输出 `canonical_url`
- 视频主卡已经输出 `capture_status`
- 视频主卡已经输出 `platform_tag / media_type / capture_method_tag / quality_tag`
- 视频主卡已经输出 `source_tags`
- 视频主卡已经输出 `transcript_method`
- 视频 fallback 卡已经输出 `fail_reason`
- 本地视频与远程视频都已纳入同一类 `video` 产物

### 8.2 差距审计（基于真实代码）

| 路径/函数 | 当前输出 | 与规范的差距 | 结论 |
|---|---|---|---|
| `write_video_note()` | 视频主卡 frontmatter 已较完整 | 缺显式 `provenance`、缺 `expanded_url`、未抽象成 Canonical Envelope 对象 | **最接近新规范，可作为基线** |
| `write_video_fallback_note()` | 有 `fail_reason`、`expanded_url`、`canonical_url`、`source_tags` | 字段名仍有 `source` / `author_fragment` 这类临时味道；与主卡不完全一致 | **fallback 逻辑正确，字段还需统一** |
| `write_note()` | 只有 `title / type / source / fetch_method` 等极简字段 | 没有 `canonical_url`、没有 `capture_status`、没有 `platform`、没有确定性标签 | **这是当前最明显的不统一点** |
| `save_metadata_only()` | 保底写卡 | 字段集偏旧，离 Canonical Envelope 还远 | **保留“失败不丢”，但必须升级结构** |
| `process_audio()` | 目前仍走 `write_note()` | 音频没有 `source_type=audio` 的正式 frontmatter | **音频链路已能处理，未正式归一化** |
| `process_link_text()` | 目前仍走通用 note | 网页/普通链接没有统一的 `canonical_url + quality + provenance` | **网页链路能跑，但规范性不足** |
| `process_local_video()` | 走 `write_video_note()` | 与远程视频大体一致 | **可直接纳入视频统一规范** |

### 8.3 审计结论

真正的问题不是“脚本不能用”，而是**只有视频链路开始接近统一模型，音频/网页/通用文本仍停留在旧 frontmatter 时代**。

所以接下来的代码改造顺序必须是：

1. 保留现有视频主卡作为新规范基线
2. 让 fallback 卡字段名向主卡靠拢
3. 淘汰或重构通用 `write_note()`
4. 让音频/网页/文档都先产出 Canonical Envelope，再映射成 staging 卡

## 九、四类输入适配矩阵

| 输入类型 | 主适配器 | 第一降级 | 第二降级 | 目标主产物 |
|---|---|---|---|---|
| 视频 | `yt-dlp` / 平台 metadata / 浏览器 metadata / ASR | 浏览器 metadata | fallback 待补抓卡 | `transcript` + video staging |
| 音频 | Whisper / faster-whisper | 元信息 + 原始文件保留 | metadata-only 卡 | `transcript` + audio staging |
| 网页 | canonical URL + 正文抽取（Jina/HTML） | metadata-only | 保底失败卡 | `markdown_body` + web staging |
| 文档 | 文本抽取器（PDF/Office/Markdown/Text） | 元信息 + 局部正文 | 保底失败卡 | `markdown_body` + doc staging |

### 9.1 每类输入的正式要求

#### 视频
- 主目标：`transcript_first`
- 优先级：字幕 > ASR > 浏览器 metadata > 待补抓
- 必留字段：`canonical_url / platform / capture_status / transcript_method`

#### 音频
- 主目标：拿到 transcript
- 若转写质量低：允许 `metadata_only`，但不能假装完整
- 后续应新增 `media_type: audio`

#### 网页
- 主目标：拿到 `markdown_body`
- 若正文抽取失败：至少留下 `source_url / canonical_url / title / fail_reason`
- 不允许只落一句“抓取失败”然后把原文丢掉

#### 文档
- 主目标：结构化正文，不是单纯 OCR 段落堆叠
- 先保层级和页内结构，再做摘要与分类

---

## 十、下一步实施要求

1. 任何新处理器都先对齐 Canonical Envelope
2. 任何新的 staging 卡 frontmatter 都必须能映射回本规范
3. 新平台接入先补平台识别规则，再写抓取逻辑
4. 先保证失败不丢，再追求全自动分类
5. 禁止继续扩散 `write_note()` 这种旧结构
