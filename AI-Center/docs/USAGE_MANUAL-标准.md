# USAGE_MANUAL 标准

> AI Center 内所有服务 / 工具 / 平台 / 基础设施的统一使用手册模板。
> 目标：每个条目都能用固定字段回答四件事：**怎么查活、怎么备份、怎么告警、怎么恢复**。

---

## 适用范围

- `services/`
- `tools/`
- `platforms/`
- `infrastructure/`

**默认放置位置**：优先放在对应 `README.md` 中；如果内容过长，再拆成同目录 `USAGE_MANUAL.md`，README 里保留链接。

---

## 强制规则

1. **字段名固定，顺序固定**，不要每个服务自己发明标题。
2. **健康检查 / 备份 / 告警 三块必须存在**；没有备份也要写明“无备份 / 不适用”。
3. **能自动采集的字段优先自动化**，例如端口监听、容器状态、HTTP health check。
4. **无法核实的信息保留原样**，不要为了完整硬补。
5. **值要可执行**：命令、路径、频率、阈值、恢复动作必须能落地。

---

## 标准模板

```md
## USAGE_MANUAL

### 健康检查

| 字段 | 值 | 说明 |
|------|----|------|
| 检查对象 | <端口 / 进程 / URL / 容器 / Cron> | 具体检查谁 |
| 检查命令 | <curl / ss / docker ps / systemctl ...> | 实际执行的命令 |
| 判据 | <200 / 端口监听 / PID 存在 / healthy> | 什么算通过 |
| 频率 | <每次变更后 / 每 5 分钟 / 每小时> | 谁来检查 |
| 失败动作 | <重启 / 告警 / 人工介入> | 检查失败后怎么处理 |

### 备份

| 字段 | 值 | 说明 |
|------|----|------|
| 备份对象 | <数据目录 / 配置 / 凭证索引 / DB> | 备什么 |
| 备份方式 | <tar / rsync / pg_dump / snapshot> | 怎么备 |
| 频率 | <hourly / daily / before upgrade> | 多久备一次 |
| 保留策略 | <7 天 / 30 份 / 轮转> | 保留多久 |
| 恢复命令 | <还原命令 / 回滚步骤> | 出事怎么恢复 |

### 告警

| 字段 | 值 | 说明 |
|------|----|------|
| 告警条件 | <health fail / disk full / backup fail> | 什么情况要通知 |
| 通知渠道 | <微信 / 飞书 / 邮件 / 终端告警> | 发给谁 |
| 兜底动作 | <自动重启 / 暂停写入 / 人工处理> | 第一响应做什么 |
| 升级路径 | <自动化优先，失败后人工> | 谁接手 |
```

## 推荐写法

### 1) 健康检查写实机证据，不写感觉

优先级建议：
1. HTTP health endpoint
2. 端口监听
3. 进程 / 容器状态
4. 日志中明确 ready 信号

### 2) 备份写成“可恢复”，不是“有备份”

必须写清：
- 备份对象在哪里
- 备份脚本做了什么
- 失败后如何恢复
- 是否纳入现有备份流水线

### 3) 告警写成“谁收到、收到后做什么”

不要只写“会通知”。要写：
- 什么触发
- 通知到哪
- 先自动修复还是先人工介入
- 是否要补一条手工确认流程

## 与服务 README 的关系

服务 README 是总入口，`USAGE_MANUAL` 是其中的固定运维段。

推荐结构：

```md
# 服务名

## 基础信息
## 架构定位
## 在本生态中的角色
## 常用命令
## USAGE_MANUAL
### 健康检查
### 备份
### 告警
### 恢复 / 回滚
## 凭证与配置
## 迭代记录
```

如果服务太简单：
- 健康检查可以只写 1 行
- 备份可以写“无独立数据，无需额外备份”
- 告警可以写“无自动告警，靠巡检”

但**三块不能缺**。
