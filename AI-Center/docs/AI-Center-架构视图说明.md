# AI Center 架构视图说明

> AI Center 文档体系采用多视图架构，不再用单张图覆盖所有内容。
> 每个视图回答不同的问题，对象可以出现在多个视图中，但只能有一个主归类。

---

## 1. 六层架构

AI Center 采用 6 层架构：

| 层级 | 名称 | 回答什么 |
|---|---|---|
| L1 | 物理层 | 东西放在哪（设备、宿主机） |
| L2 | 连通层 | 东西怎么连（路由、VPN、局域网） |
| L3 | 运行层 | 系统里实际跑什么（服务、进程） |
| L4 | 能力层 | 系统能提供什么可复用能力 |
| L5 | 项目层 | 现在在做什么事（工作包） |
| L6 | 项目状态层 | 下次捡起来时从哪里继续 |

### 依赖方向

```text
L5 项目层 → L4 能力层 → L3 运行层 → L2 连通层 → L1 物理层
```

依赖只能向下，不能反向。

## 3. 中台与项目的边界

### 中台（能力层）
- LiteLLM = 模型路由中台
- Hindsight = 记忆/检索中台
- n8n = 自动化编排中台
- 通知链路 = 告警中台
- Hermes / OpenClaw = 入口与执行控制面

### 项目（项目层）
- 某个单次迁移任务
- 某个服务的临时改造
- 某次模型上线/下线
- 某个网络拓扑调整

**判断标准**：可复用、被多个项目消费的才算中台；单次维护、迁移、上线一律是项目。

## 4. 文档与记忆的分工

| 维度 | 文档负责 | 记忆负责 |
|---|---|---|
| 进度 / todo | ✅ | ❌ |
| checkpoint / 决策记录 | ✅ | ❌ |
| 接手摘要 | ✅ | ❌ |
| 用户长期偏好 | ❌ | ✅ |
| 稳定规则 / 共识 | ❌ | ✅ |
| 重复踩坑经验 | ❌ | ✅ |

**项目状态不能只存在记忆里，必须有一张可恢复的状态卡。**

## 4. Harness 治理层（横切维度）

Harness 不是第 7 层，而是横切进 6 层架构的治理协议。

**Agent = Model + Harness。** Model 是大脑，Harness 是围绕大脑的身体+工具+记忆+护栏+反馈。

### 8 构件

| # | 构件 | 解决的问题 |
|---|---|---|
| 1 | System Prompt | 行为边界定义 |
| 2 | Tools | 让模型能动手 |
| 3 | Sandbox | 隔离、可重置 |
| 4 | Filesystem | 跨会话不丢 |
| 5 | Memory | 短期压缩 + 长期检索 |
| 6 | Feedback | 自动验证任务完成 |
| 7 | Guardrails | 危险动作人审 |
| 8 | Observability | 日志可回放 |

### 合规度红线
- < 4/8：禁止上线
- 4-5/8：小范围试点
- 6/8：灰度上线
- 7-8/8：正式上线

## 5. 主归类规则

一个对象可以出现在多个视图中，但只能有一个主归类。

| 对象 | 主归类 | 次归类 | 备注 |
|---|---|---|---|
| Hermes | 运行层 | 入口/控制面 | 既是执行服务，也是控制入口 |
| OpenClaw | 运行层 | 入口/控制面 | 与 Hermes 形成执行链路 |
| LiteLLM | 运行层 | 能力层（模型路由） | 典型中台能力 |
| Hindsight | 运行层 | 能力层（记忆/检索） | 典型中台能力 |
| Homepage | 运行层 | 门户/观察面 | 展示状态，不是事实源 |
