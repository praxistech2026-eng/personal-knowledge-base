# 知识摄入样例验证（2026-06-11）

> 这不是理论推演，是基于现有 `_staging/` 真实卡片回读得出的验证结果。

---

## 一、验证目的

验证三件事：

1. 新四层架构能否覆盖真实输入
2. staging frontmatter 是否已经开始向 Canonical Envelope 对齐
3. 哪些平台已经够稳，哪些还只是“能产出保底卡”

## 三、样例结论

## 3.1 YouTube 样例

### 已验证

- 有 `source_url`
- 有 `canonical_url`
- 有 `platform / platform_tag`
- 有 `media_type`
- 有 `capture_method_tag`
- 有 `quality_tag`
- 有 `source_tags`
- 有 `video_id`
- 有 `author / duration / transcript_method`
- 有正文级 transcript

### 判断

这是当前最接近新规范的样例。

### 问题

- `capture_method_tag=asr`，说明这次不是优质原生字幕路径
- transcript 文本质量一般，后续仍要优先争取平台字幕

---

## 3.2 B站样例

### 已验证

- 有 `source_url / canonical_url`
- 有 `platform`
- 有 `video_id / author / duration`
- `capture_status=metadata_only`

### 问题

- 缺 `platform_tag / media_type / capture_method_tag / quality_tag / source_tags`
- 说明老卡与新规范并存

### 判断

B站链路能产出，但字段尚未完全统一到最新 schema。

---

## 3.3 抖音样例

### 已验证

- 有 `canonical_url`
- 有 `capture_status=metadata_only`
- 有平台、作者、时长、点赞片段
- 明确保留了 `browser-metadata-only` 失败上下文

### 判断

抖音链路的现实策略是对的：**拿不到视频正文时，至少保住 metadata 和 canonical URL。**

### 问题

- 仍缺 `platform_tag / media_type / capture_method_tag / quality_tag / source_tags`
- 说明主卡模板升级后，旧样例还没回填

---

## 3.4 小红书样例

### 已验证

- 有 `platform_tag / media_type / capture_method_tag / quality_tag / source_tags`
- 有 `canonical_url / expanded_url`
- 有 `fail_reason`
- `capture_status=manual_review`

### 判断

小红书 fallback 卡反而更符合“失败不丢 + 可重抓”的新规范。

### 问题

- 仍有 `source`、`author_fragment` 这类旧字段命名
- 说明 fallback 结构对了，但字段名还没清洗干净

---

## 四、总体判断

### 已被证明成立的部分

1. 新四层架构能覆盖真实视频输入
2. `canonical_url + capture_status + source_tags` 这条路是对的
3. `manual_review / metadata_only / transcript_first` 这组质量分层是实用的
4. fallback 卡必须存在，这一点已经被现实验证

### 暴露出来的真实问题

1. 视频新卡与旧卡并存，字段不一致
2. 音频 / 网页 / 文档样例还没系统回填
3. 通用 `write_note()` 仍代表旧时代结构

## 六、结论

现在可以明确说：

- 你的知识库不是“完全没法用”
- 它的问题也不是“没换成 Codex”
- 真问题是：**只有视频链路的一部分已经进入新架构，其他类型和旧卡结构还没被统一**
