## 1. 背景与目标

### 1.1 问题陈述

当前系统已接入多厂商模型套餐：

| 厂商 | 套餐 | 配额特征 |
|---|---|---|
| MiniMax | Starter ¥29/月 | 600次/5h，个人专属，无周月限 |
| 百炼 | Pro ¥200/月 | 6000次/5h，月9万次，共用账号 |
| 已下架厂商 | Max ¥469/月 | ~1600 prompts/5h，周8000次，共用账号 |
| SenseNova | 限时免费 | 1500次/5h（自有模型），150次（DeepSeek），随时可能结束 |
| ChatGPT | Plus $20/月 | 30-150条/5h（共享额度），全系共享同一池 |
| Gemini | AI Studio Tier1 付费 | RPM/RPD 速率限制，无次数上限 |

**核心问题**：

1. 模型多、套餐复杂，没有统一调用策略
2. 手动切换模型繁琐，容易"用完贵的才发现大量简单任务用了旗舰"
3. 各厂商配额限制不统一，无法用同一套逻辑评估

### 1.2 目标

建立一套**可落地、可拓展**的模型评价与调用体系，实现：

- 自动判断任务类型、复杂度、价值
- 自动路由到合适的模型
- 在质量、成本、配额三者间取得平衡

## 2. 分层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    你的输入（说话）                         │
└─────────────────────────┬───────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  应用层：Hermes                                             │
│  ├── 判断 task_type（任务类型）                             │
│  ├── 判断 complexity（复杂度）— 规则 + LLM辅助             │
│  ├── 判断 value（价值）— 规则                              │
│  └── 组合决策 → 确定 model_name + order                     │
└─────────────────────────┬───────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  执行层：LiteLLM                                            │
│  ├── 接收 model_name（default/coding/reasoning/vision/...） │
│  ├── 按 order 顺序执行（order 1 → order 2 → order 3）       │
│  ├── 429/错误 → 自动切换下一个 order                       │
│  └── 全挂 → fallback 到其他 model_name 组                  │
└─────────────────────────┬───────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  厂商层                                                     │
│  MiniMax / 百炼 / 智谱 / SenseNova / ChatGPT / Gemini      │
└─────────────────────────────────────────────────────────────┘
```

**分工原则**：

- **LiteLLM 只做执行**：按 order 顺序执行请求，处理 429 fallback，不动脑子判断
- **判断逻辑在 Hermes（应用层）**：task_type / complexity / value 全在应用层判断

## 3. 评价体系：三维度评分

### 3.1 三维度定义

| 维度 | 含义 | 决定什么 |
|---|---|---|
| **Q（质量）** | 模型能力的高低 | 选哪个 model_name 组，用组内哪个 order |
| **C（成本）** | 每次调用消耗钱包的痛感 | 同样的任务，用贵模型还是便宜模型 |
| **P（优先级）** | 厂商/套餐的节制程度 | 是否应该用这个模型，应该用多少 |

### 3.3 成本分 C（成本痛感，0-100，越高=越不心疼）

成本分衡量"这次调用消耗你钱包的痛感"，不是单纯的 API 单价。

**核心原则**：

- 免费/不额外花钱 = C 最高
- 共用账号（百炼/智谱）虽然共用，但不额外扣钱，C 应该高
- 真正的成本限制交给 P（优先级）去控

| 模型群 | C_score | 理由 |
|---|---|---|
| SenseNova 全系 | **100** | 完全免费，随便用 |
| GLM 全系（智谱） | **90** | 共用账号，不额外花钱，配额节制靠 P |
| 百炼 全系 | **90** | 共用账号，不额外花钱，配额节制靠 P |
| MiniMax-M2.7 | **80** | 个人付费，600次/5h 无周月限 |
| Gemini-Flash-Lite | **75** | 个人付费，RPD 500，量大 |
| Gemini-Flash（非Lite） | **50** | 个人付费，RPD 相对紧 |
| Gemini-Pro | **40** | 个人付费，RPD 100 极紧 |
| ChatGPT-5.4-mini | **60** | 个人付费，限额较松（60-350/3h） |
| ChatGPT-Codex | **45** | 个人付费，限额紧（30-150/5h） |
| ChatGPT-5.4/5.5 | **25** | 个人付费，限额极紧（20-100/3h） |

### 3.4 优先级分 P（厂商优先级，1.0-0.3）

优先级反映厂商/套餐的获取难度和节制必要性。

| 档位 | P_score | 厂商 | 理由 |
|---|---|---|---|
| 免费，白嫖优先 | **1.0** | SenseNova | 完全不花钱，随便用 |
| 个人付费，放心用 | **0.85** | MiniMax | 个人专属套餐，无周月限 |
| 个人付费，可用但要节制 | **0.80** | Gemini | 个人付费，但 RPD 有硬限 |
| 个人付费，额度紧 | **0.70** | ChatGPT | 个人付费，但共享 5h 窗口额度和周限额 |
| 共用，需节制 | **0.50** | 百炼 | 月 9 万次，共用账号，不能当主力 |
| 共用，高度节制 | **0.40** | 已下架厂商 | 周 8000 次，共用账号+有倍率消耗 |

## 4. 任务分类体系

### 4.1 任务类型 task_type（决定用哪个 model_name 组）

| task_type | LiteLLM model_name | 判断关键词 |
|---|---|---|
| **coding** | `coding` | 写代码/改代码/调试/重构/解释代码/debug/函数/程序 |
| **text** | `default` | 对话/问答/解释/总结/翻译/润色/写文章/写文案 |
| **reasoning** | `reasoning` | 分析/推理/证明/数学/逻辑/方案设计/评估 |
| **vision** | `vision` | 看图/图片理解/分析图片/解读图片 |
| **image** | `image` | 生成图片/画图/生图/作图 |
| **agent** | 多模型组合 | 拆解任务/多步骤/并行执行/规划（由 Agent 层处理） |

**判断方式**：关键词匹配，优先规则判断，不命中的由 Hermes 进一步分析或向用户确认。

### 4.2 任务复杂度 complexity（决定在 model_name 组内选哪个 order）

| 等级 | 含义 | 判断规则 |
|---|---|---|
| **L1 简单** | 单次调用，单一输出 | 包含"单文件"/"单个"/"简单"/"写一个函数"/"帮我看看这段代码"/"翻译一下"/"总结这篇" |
| **L3 复杂** | 5+次调用，开放性 | 包含"系统"/"架构"/"设计"/"重构"/"重写"/"迁移"/"微服务"/"分布式"/"多文件"/"跨文件"/"模块化" |
| **L2 中等** | 2-5次调用，兜底 | 不符合 L1 也不符合 L3 → 默认 L2 |

### 4.3 任务价值 value（决定是否要用高质量模型）

| 等级 | 含义 | 判断规则 |
|---|---|---|
| **V3 高价值** | 正式场合/生产环境/不可失败 | 包含"上线"/"生产"/"正式"/"用户面向"/"方案"/"报告"/"重要"/"关键"/"618"/"双11"/"竞标"/"汇报"/"老板" |
| **V1 低价值** | 练手/一次性/错了无所谓 | 包含"demo"/"练习"/"试试"/"随便"/"学习"/"练手"/"todolist"/"备忘录"/"贪吃蛇"/"2048"/"个人用"/"自己看" |
| **V2 中价值** | 日常工作/有参考价值 | 不符合 V1 也不符合 V3 → 默认 V2 |

## 6. 判断不了时怎么办

### 6.1 触发条件

- 输入太短（少于 5 个字）
- 关键词都不匹配
- 歧义太大（"帮我写个"后面没了）

### 6.2 处理方式

```
无法判断 → Hermes 向用户提问：
"我想帮你，但需要确认一下：你想做什么类型的任务？"
选项：写代码 / 写文章 / 看图 / 生图 / 其他

用户回复后 → 记录本次判断结果 → 下次同类输入自动应用
```

### 6.3 用户修正（当判断错了）

```
模型自动路由到某模型
用户说："这是 coding 任务，而且是个简单 demo"
→ Hermes 记住：xxx 关键词 → coding/L1/V1
→ 下次同类请求自动应用该判断
```

## 9. LiteLLM 的能力与局限

### 9.1 LiteLLM 能做什么

- **model_name 分组**：default / coding / reasoning / vision / image / cheap
- **order 排序**：按你定的顺序执行，429 自动切换下一个
- **fallback 链**：model_name 组全挂 → 自动切到另一个 model_name 组
- **RPM/TPM 限制**：硬限制触发限流
- **健康检查**：后台自动检测模型可用性

### 9.2 LiteLLM 不能做什么

- 按 Q/C/P 系数自动排序（必须手动写 order）
- 按复杂度/价值动态调整 order
- 自动判断任务类型（这在 Hermes 层做）
- 跨厂商智能路由（只能做可靠性 fallback）

### 9.3 实际分工

```
LiteLLM = 听话的执行者
  - 接收 Hermes 的指令（用哪个 model_name）
  - 按 order 执行
  - 处理 429 / 错误 fallback

Hermes = 动脑的判断者
  - 判断 task_type
  - 判断 complexity
  - 判断 value
  - 决定用哪个 model_name
```
