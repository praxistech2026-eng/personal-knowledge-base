# 2. 最大风险：双调度器问题

我前面说：

```text
Hermes = 总控
OpenClaw Manager = 内部调度
```

这个说法容易有隐患。

因为严格来说，这里已经有两个“像调度器”的东西：

1. Hermes
2. OpenClaw Manager

如果边界不清楚，长期一定乱。

---

## 正确修正

应该改成：

```text
Hermes = 决策控制器 / 用户代理 / 最终裁决者
OpenClaw Manager = 执行调度器 / 队列管理员 / 状态推进器
```

也就是说：

## Hermes 可以决定：
- 这个任务要不要启动
- 是不是需要 OpenClaw 团队
- 是否允许低成本 worker 参与
- 最终结果是否可以交付给你
- 是否需要更新长期记忆或拓扑图

## OpenClaw Manager 只能决定：
- 这个已批准任务分给哪个 worker
- worker 是否完成
- 是否进入 review
- 是否需要重试
- 是否上报 Hermes

### OpenClaw Manager 不应该决定：
- 是否改变目标
- 是否扩大任务范围
- 是否做架构决策
- 是否跳过关键 review
- 是否修改全局规则

一句话：

> **Hermes 管“该不该做、做到什么程度”。  
> OpenClaw 管“已经决定要做的事怎么分派执行”。**

# 3. 第二个风险：不能完全依赖“智能分配”

你希望有管理型 Agent 智能分配任务。这个方向没问题，但不能让它完全自由判断。

否则长期会出现：

- 今天这个任务给 Context Fetcher
- 明天同类任务给 Executor
- 有时候低成本模型越权做判断
- 有时候 review 被跳过
- 有时候任务被拆得太碎

---

## 建议：规则优先，LLM 辅助

应该做成：

```text
先用固定规则路由
规则无法判断时，才让高质量模型裁决
```

也就是：

### 固定规则示例

| 任务类型 | 默认角色 | 可用模型 |
|---|---|---|
| 查询资料 | Context Fetcher | 低成本 |
| 提取上下文 | Context Fetcher | 低成本 |
| 文档格式整理 | Reporter / Executor | 低成本 |
| 明确模板填充 | Executor | 低成本 |
| 代码修改 | Executor | 中高质量 |
| 架构设计 | Hermes / Architect | 高质量 |
| 风险判断 | Hermes / Reviewer | 高质量 |
| 最终定稿 | Hermes | 主模型 |
| 拓扑图更新 | Executor + Reviewer + Hermes 审核 | 中高质量 |

### 只有这些情况才允许 Manager “智能判断”
- 任务类型不明确
- 涉及多个角色
- worker 反馈阻塞
- review 发现问题
- 任务可能越权
- 涉及凭证、安全、系统配置

# 5. 第四个风险：低成本模型需要“硬边界”

我前面说低成本模型只做机械任务，这是对的。  
但光靠 prompt 约束不够。

必须有三层边界：

## 第一层：任务类型边界
低成本模型只能接：

- fetch_context
- summarize_sources
- format_existing_content
- template_fill
- status_report
- archive

不能接：

- decide
- design
- refactor
- security_review
- deploy
- config_write
- memory_update
- topology_update_final

---

## 第二层：工具权限边界

低成本 worker 尽量只给：

- web/search
- read-only file
- 写入指定 shared/context 或 shared/artifacts 的权限

不要给：

- shell unrestricted
- patch 任意文件
- 配置文件写权限
- env 文件读取权限
- 拓扑图直接写权限
- gateway restart 权限

---

## 第三层：输出格式边界

低成本 worker 输出必须是这种形式：

```markdown
# Context Report

## Sources
- [url] title, accessed_at

## Extracted Facts
- fact 1
- fact 2

## Unknowns
- unknown 1

## Not Performed
- No recommendation made
- No config changed
- No final decision made
```

强制它声明：

> 我没有做判断，我没有改配置，我没有做最终建议。

# 6. 第五个风险：review 不应该“一刀切”

前面我说 review 不可省。方向对，但实际要分级，否则会太慢。

---

## 建议 review 分三级

### L1：轻量 review
适合：
- 查询
- 摘要
- 格式整理
- 简单归档

检查：
- 是否有来源
- 是否缺字段
- 是否越权下结论

可以由便宜 reviewer 或规则检查完成。

---

### L2：标准 review
适合：
- 文档更新
- 拓扑图更新
- 使用说明
- 配置草案

检查：
- 是否符合事实
- 是否和现有文档冲突
- 是否泄露凭证
- 是否能被用户执行

---

### L3：强 review
适合：
- 修改 Hermes/OpenClaw 配置
- 写入 `.env`
- 改 gateway
- 加 cron
- 改服务启动方式
- 权限/凭证相关操作

要求：
- 高质量模型
- 明确 diff
- 必要时询问你确认
- 执行前后验证

```yaml

```

Hermes 读取结果、review、decision，再回复你。

- 当前 skills 是否可用
- 当前 OpenClaw 插件是否可用

否则新拓扑会建立在旧事实上。

# 19. 我的最终建议

保留原方向，但把方案升级为：

```text

  ↓
Policy Router
  ↓
OpenClaw Manager
  ↓
Task Ledger
  ↓
Specialized Workers
  ↓
Review Gate
  ↓
Hermes Finalization
```

这比我前面说的：

```text
Hermes → OpenClaw → workers
```

要稳定得多。

一句话：

> **不要搭一个“多 agent 聊天团队”。  
> 要搭一个“有任务账本、有权限边界、有审查闸门的 agent runtime”。**

这才符合你要的：长期、低出错、高兼容。

### [] user

这么说，是要长时间迭代的，那么迭代的机制有设计吗

### [] assistant

有，必须设计。否则这套系统很容易变成：

> 初期看起来很聪明，跑几周后规则漂移、角色混乱、文档过期、任务状态不可追踪。

所以我的建议不是“一次搭完”，而是设计成一个**长期可控迭代系统**。

核心原则：

> **允许系统进化，但不允许系统自己乱进化。**  
> **所有迭代都要有记录、有评审、有回滚、有版本。**

# 1. 总体迭代机制

我建议把 AI Center Agent Team Runtime 的迭代机制设计成三层闭环：

```text
日常任务闭环
  ↓
质量改进闭环
  ↓
架构演进闭环
```

对应三种不同粒度：

| 层级 | 目的 | 频率 | 谁主导 |
|---|---|---|---|
| 任务级迭代 | 修正单个任务中的问题 | 每个任务完成后 | OpenClaw Manager + Reviewer |
| 系统级迭代 | 改进 agent 角色、规则、模板 | 每周或每 10 个任务 | Hermes |
| 架构级迭代 | 调整整体架构、模型策略、权限边界 | 每月或重大故障后 | Hermes + 你确认 |

---

# 2. 不能让 agent 自己无限自我优化

这个点很重要。

我不建议做成：

```text
agent 发现自己不好 → 自己改 prompt → 自己上线 → 自己继续跑
```

这很危险。

因为它会导致：

- 角色边界漂移
- 越权越来越多
- prompt 越改越复杂
- 问题来源不可追踪
- 后面没人知道为什么变成这样

---

## 正确机制应该是

```text
发现问题
  ↓
记录证据
  ↓
提出改进建议
  ↓
Review
  ↓
Hermes 或你批准
  ↓
进入灰度
  ↓
验证有效
  ↓
正式生效
```

也就是：

> agent 可以**建议迭代**，但不能**擅自生效**。

---

# 3. 迭代主循环：Observe → Evaluate → Propose → Approve → Apply → Verify → Document

我建议所有长期改进都走这个固定循环。

```text
1. Observe   观察问题
2. Evaluate  评估问题
3. Propose   提出改进
4. Approve   批准变更
5. Apply     应用变更
6. Verify    验证效果
7. Document  写入文档
```

---

## 3.1 Observe：观察

系统持续记录：

- 哪些任务失败
- 哪些任务超时
- 哪些 worker 越权
- 哪些 review 退回
- 哪些任务需要 Hermes 接管
- 哪些任务重复出现
- 哪些 prompt 约束没生效

这些不是靠感觉，而是写进 task ledger。

示例：

```yaml
observations:
  - time: 2026-05-05T14:00:00+08:00
    task: TASK-20260505-003
    issue: context-fetcher 给出了方案建议，越权
    severity: medium
    evidence: /shared/reviews/TASK-20260505-003-review.md
```

---

## 3.2 Evaluate：评估

不是所有问题都值得改系统。

要区分：

| 问题类型 | 处理方式 |
|---|---|
| 单次偶发 | 记录即可 |
| 同类问题出现 2-3 次 | 修改模板或 prompt |
| 影响安全/凭证/配置 | 立即修规则 |
| 影响最终交付质量 | 加强 review |
| 影响架构边界 | 做 ADR 决策 |

---

## 3.3 Propose：提出改进

改进建议必须写成 proposal，不允许口头改。

建议格式：

```markdown
# PROPOSAL-20260505-001: 限制 Context Fetcher 输出建议性语言

## 背景
Context Fetcher 在 TASK-xxx 中给出了配置建议，违反“只整理上下文”的边界。

## 证据
- /shared/reviews/TASK-xxx-review.md

## 建议修改
在 Context Fetcher SOUL.md 中增加：
“不得使用建议、推荐、应该、最好等决策性表达。”

## 影响范围
- Context Fetcher
- TASK_PROTOCOL.md 输出模板

## 风险
可能导致摘要偏保守，但符合低成本 worker 定位。

## 验证方式
连续 5 个 fetch_context 任务中不得出现决策性表达。
```

---

## 3.4 Approve：批准

批准规则要分级。

| 改动类型 | 谁批准 |
|---|---|
| 修正文案模板 | Hermes 可批准 |
| 修改低成本 worker prompt | Hermes 可批准 |
| 修改角色边界 | Hermes + Reviewer |
| 修改权限/工具范围 | 你确认 |
| 修改服务配置、密钥、网关 | 必须你确认 |
| 架构变更 | 必须你确认 |

---

## 3.5 Apply：应用

应用时不能直接覆盖旧版本。

必须：

- 备份旧版本
- 写 changelog
- 标记版本号
- 小范围试用

---

## 3.6 Verify：验证

改完后必须验证是否真的改善。

例如：

```yaml
verification:
  sample_size: 5
  metric:
    - low_cost_worker_overreach_count == 0
    - review_return_rate <= previous_rate
    - no_new_security_warning
```

---

## 3.7 Document：文档化

验证通过后，才写入：

- 拓扑图
- 使用说明
- agent role 文档
- task protocol
- changelog

# 7. 变更类型

长期迭代主要分 6 类。

---

## 7.1 Prompt 迭代

比如：

- 低成本 worker 老是越权
- reviewer 检查不够严
- reporter 太啰嗦
- manager 分派不稳定

处理方式：

```text
记录问题 → 修改 SOUL.md → 小样本测试 → review → 生效
```

---

## 7.2 Policy 迭代

比如：

- 哪些任务能用低成本模型
- 哪些任务必须 review
- 哪些任务必须问你确认
- 哪些工具能给哪个 agent

这个写在：

```text
POLICY_ROUTER.md
```

Policy 变更比 prompt 变更更敏感，需要更严格 review。

---

## 7.3 Workflow 迭代

比如：

- 增加 Cancel 状态
- 增加 Blocked 状态
- 修改 retry 机制
- 修改 handoff 模板

这个写在：

```text
TASK_PROTOCOL.md
```

---

## 7.4 Agent 角色迭代

比如新增：

- Security Reviewer
- Topology Maintainer
- Research Synthesizer
- Knowledge Curator

必须先写角色定义，不允许直接上线。

---

## 7.5 Model 策略迭代

比如：

- SenseNova 只用于 fetch_context
- MiniMax-M2.7 用于 Hermes 主控
- GPT-5.4 只用于高价值裁决
- Doubao 作为 fallback

这属于成本与质量策略，需要记录。

---

## 7.6 架构迭代

比如：

- 从文件队列改为 SQLite
- 从轮询改为 webhook
- 引入长期任务 daemon
- 引入可视化任务面板

这种必须写 ADR。

# 8. ADR：架构决策记录

凡是“以后别人会问为什么这么做”的，都写 ADR。

示例：

```markdown
# ADR-001: 使用文件化 Task Ledger 而不是数据库

## 状态
Accepted

## 背景
AI Center 初期需要低复杂度、高可审计、易恢复的多 agent 状态层。

## 决策
第一版使用 YAML task files + jsonl event log。

## 备选方案
1. SQLite
2. Kanban DB
3. OpenClaw 内部状态
4. 纯聊天上下文

## 理由
文件方式最透明、最容易人工修复、最适合早期迭代。

## 代价
并发控制弱，需要 lock 文件或单 manager 领取机制。

## 触发重新评估条件
- 任务数超过 500
- 并发 worker 超过 5
- 出现重复领取问题超过 2 次
```

没有指标，迭代就会变成感觉。

## 9.1 核心指标

| 指标 | 含义 |
|---|---|

| review_return_rate | review 退回率 |
| low_cost_overreach_count | 低成本 worker 越权次数 |
| missing_artifact_count | 产物丢失次数 |
| stuck_task_count | 卡住任务数量 |
| avg_task_duration | 平均任务耗时 |

```yaml
  stuck_task_count: "<= 1 per week"
  topology_drift_review: "weekly"
```

# 10. 迭代节奏

我建议分三种节奏。

---

## 10.1 每任务结束后

做轻量复盘：

```text
任务是否完成？
是否需要 review 退回？
worker 是否越权？
产物是否在正确位置？
是否需要更新规则？
```

不是每次都改规则，只记录。

---

## 10.2 每周一次

系统复盘：

```text
过去一周失败最多的任务类型是什么？
```

# 17. 最重要的防漂移设计

长期多 agent 系统最大敌人是漂移。

---

## 17.1 角色防漂移

每个 agent 都有 SOUL.md。

不能直接口头改角色。  
所有角色变更必须更新 SOUL.md 和 CHANGELOG。

---

## 17.2 Policy 防漂移

所有路由规则写在 POLICY_ROUTER.md。

Manager 不能凭感觉绕过 policy。

---

## 17.3 状态防漂移

所有任务状态以 task yaml 为准。

聊天记录不作为权威状态。

---

## 17.4 文档防漂移

拓扑图定期 Reality Audit。

如果文档和实际冲突，以实际审计结果为准，然后修文档。

---

## 17.5 质量防漂移

每周检查 review_return_rate、越权次数、失败任务。

一旦恶化，就暂停扩展，先修系统。
