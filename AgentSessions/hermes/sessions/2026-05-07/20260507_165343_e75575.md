## 为什么 TikTok 不适合地域故转？

地域故转的逻辑是：

```text
手动节点不可用
↓
自动切到另一个可用节点
↓
手动恢复后切回
```

这适合普通地域流量，比如：

```text
美国网页
日本网页
新加坡普通代理
```

但 TikTok 不一样。

TikTok 更敏感的是：

```text
账号地区一致性
内容推荐地区一致性
SIM / 设备 / IP 地区一致性
```

如果 TikTok 今天走日本，明天自动切新加坡，后天又切美国，可能导致：

- 推荐流混乱；
- 账号地区判断不稳定；
- 运营账号风控；
- 内容受众地区漂移；
- 账号行为看起来像异地异常。

所以 TikTok 不应该跨国家自动故转。

## 为什么里面还是选 `日本-故转 / 新加坡-故转`？

这里的“故转”只在**同一地区内部**兜底。


实际是：

```text
TikTok

日本-故转
↓
日本-手动

日本-自动
```

它只会在日本节点内部兜底，不会自动跳到新加坡/美国。

这就符合 TikTok 的需要：

```text
国家固定
节点可换
地区不乱跳
```

## 2. 所以“网易走新加坡”的真实可能原因

按概率排序：

---

### 原因 A：当时运行的不是 v14，而是旧配置

这是最大可能。

网易新加坡提醒发生时，你大概率还在跑 v9 / v10 / v11 / v12 / v13 中某个旧版本，或者 OpenClash 没真正 reload 成新版本。

尤其如果当时日志里没有类似：

```text
match RuleSet/NetEase
match DomainSuffix/163.com
```

而是走了：

```text
MATCH → 🌐 通用兜底 → 新加坡
```

那说明当时的规则没有拦住。

v14 是后面才加固的。

---

### 原因 B：OpenClash 当时在 GLOBAL / 全局模式

如果模式不是：

```text
Rule / 规则模式
```

而是：

```text
Global / 全局模式
```

那所有国内网站都可能走你当时选中的新加坡节点。

这种情况下：

```yaml
China-Domain
NetEase
Direct
GFW
```

这些规则都不发挥正常作用。

---

### 原因 C：规则 provider 没加载成功

如果 `China-Domain`、`Direct`、`NetEase` 规则源下载失败，规则本身不存在，流量就会往后掉。

但 v14 里我加了手写：

```yaml
DOMAIN-SUFFIX,163.com,直连
DOMAIN-SUFFIX,netease.com,直连
```

这类规则不依赖 provider。  
所以如果导入 v14 后还走新加坡，就不是 provider 没下载这么简单，而是配置没加载 / 全局模式 / 流量没经过域名规则。

---

### 原因 D：网易邮箱客户端不是用域名连接，而是直接连 IP

如果某个 App 或系统先在 Clash 外部解析 DNS，然后直接拿 IP 去连，Clash 日志里可能看不到：

```text
mail.163.com
```

只看到：

```text
IP:443
```

这种情况下域名规则不会命中，只能靠：

```yaml
GEOIP,CN,直连
```

v14 已经加了：

```yaml
- GEOIP,CN,直连
```

放在 `MATCH` 前面，就是为了解决这种问题。

但如果这个 IP 是网易使用的海外 CDN / 海外风控接口，那么 `GEOIP,CN` 也拦不住，会进入 `MATCH → 通用兜底`。

这种情况需要看具体日志里的目标域名或目标 IP。

---

### 原因 E：设备上还有 App 级代理 / 浏览器代理

例如手机、电脑、浏览器、邮件客户端自己设置了 SOCKS/HTTP 代理，或者某个软件不走路由器透明代理，而是直接走：

```text
192.168.2.x → Clash mixed-port / socks-port → 新加坡节点
```

这种情况下 OpenClash 的规则可能仍然生效，但如果你当时手动选择的是某个新加坡节点作为全局出口，也可能触发网易提醒。

# 2. 为什么美国自动会显示 40 多 ms？

从你北京家宽到真正美国出口，**40ms 不合理**。  
真实北京到美国常见应该是：

```text
160ms - 220ms+
```

所以美国显示 40ms，通常不是“你到美国只有 40ms”，而是下面几种情况之一：

---

## 原因 1：节点名写美国，但实际出口不是美国

机场节点名字不一定可信。

比如节点叫：

```text
美国 01
US Los Angeles
美区 IPLC
```

但实际可能是：

```text
入口在香港 / 日本 / 新加坡
出口不是美国
或者机场中转标注混乱
```

Clash 的自动测速只看这个节点访问测速 URL 的延迟，不验证它是不是真美国 IP。

---

## 原因 2：测速 URL 是 CDN / Anycast，不代表目标业务

当前测速 URL 是：

```text
https://www.gstatic.com/generate_204
```

这个 URL 适合判断“节点活着没有”，但不适合判断：

```text
真实美国落地
真实 ChatGPT 可用性
真实 Netflix 解锁
真实 TikTok 区域
```

它可能命中 CDN / Anycast 边缘，导致延迟看起来异常低。

---

## 原因 3：测速测的是“健康检查可达”，不是地理位置

`url-test` 的目标是：

```text
谁最快，选谁
```

不是：

```text
谁是真的美国，选谁
```

所以：

```text
美国-自动 40ms
```

只能说明：

```text
某个被美国 filter 匹配到的节点，对 gstatic 测速很快
```

不能证明：

```text
它是真美国出口
```

有些机场节点是：

```text
国内 / 香港入口
↓
专线中转
↓
海外出口
```

测速显示有时接近入口延迟，不等于目标网站最终体验一定一样。

`url-test` 自动组会把符合筛选条件的节点都纳入候选池。

```text
维护节点
名字误匹配节点
```

```text

不可用 → 显示超时 / 红色
```

所以“自动组里有不可用节点”不是说它一定会选不可用节点，而是说明候选池不干净。

# 8. 你现在应该怎么看自动延迟？

不要把自动延迟当成“真实物理距离”。

应该这样理解：

| 现象 | 正确解释 |
|---|---|
| 美国 40ms | 高概率不是真美国物理链路，可能是误标 / 中转 / CDN 测速假象 |
| 自动组有红色节点 | 候选池里有不可用节点，url-test 会避开它们 |
| 某个自动节点延迟很低 | 只能说明它访问测速 URL 快，不代表 ChatGPT / TikTok / Netflix 快 |
| 自动组选错国家 | 说明 filter 误匹配或节点名不可信 |
| 所有-自动乱跳 | 全池自动天然不稳定，不适合作默认兜底 |
