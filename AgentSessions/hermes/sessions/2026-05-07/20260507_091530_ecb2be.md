OpenClash 节点 IP 查询的几种方法：

## 1. 查看配置文件

OpenClash 的节点配置通常在：
```
/etc/openclash/config.yaml
```
或者 dashboard 中的「配置管理」里直接查看。

搜索 `server:` 字段，每个节点会显示类似：
```yaml
- name: "节点A"
  type: ss
  server: 1.2.3.4
  port: 443
```

## 2. OpenClash 面板查看

在 OpenWrt Web UI → 高级设置 → OpenClash → 流量配置 → 节点列表，可以看到所有节点名称和服务器地址。

## 3. 主动测试当前出口 IP

如果想验证「我现在用这个节点出去，IP 是多少」：

```bash
# 切换到某个节点后，执行
curl -s https://api.ipify.org
```

你用的哪个节点，IP 就是节点对应服务器的 IP。

## 4. 批量获取节点 IP

如果配置里是域名而非 IP，可以用 nslookup 批量解析：
```bash
grep "server:" /etc/openclash/config.yaml | awk '{print $2}' | xargs -I {} nslookup {}
```

# 方案一：用 OCI Run Command 注入公钥【优先】

这是最安全的方式，不需要 SSH，不需要停机。

在 Oracle 控制台：

```text
Compute → Instances → Praxis → Resources → Run command
```

或者看：

```text
Oracle Cloud Agent → Plugins → Compute Instance Run Command
```

如果这个插件是 Enabled / Running，就可以直接运行命令。

运行内容用这个：

```bash
#!/bin/bash

USER_NAME=ubuntu

mkdir -p /home/${USER_NAME}/.ssh

cat >> /home/${USER_NAME}/.ssh/authorized_keys <<'EOF'
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQC0Ijfmc3dB6CouEQ8P0pwQDk7us+RYUheoGf66WPOb7ag3fz9DDEl8rMZNH2R1bThAZte+LSLbNsvjTm6f2xPR3WSvSs0w3aOSkqPkRxGcTscBQdI93TiiyvRBHTrSPc+Dj6FDZhBk7wkJKNCCSQ2VR1iNcNKCyWbmI17U5KNSuEpEYIMImciFG52KrDvR6Z/9JxRWngXAzq6XehWqFdYA3Dbqh0qdZAylp8/+Qivy315++TyrOhRKoYo0NUZq1m6FvnA6oGJ977SaX0rzhl/OxeOx+QwbZ1i6DHELu8lXjsthoKl2i+PPpiap84ZWlYWZQjl2Zk5Thv5CtliHpFZP ssh-key-2026-05-06-2
EOF

chown -R ${USER_NAME}:${USER_NAME} /home/${USER_NAME}/.ssh
chmod 700 /home/${USER_NAME}/.ssh
chmod 600 /home/${USER_NAME}/.ssh/authorized_keys

systemctl restart ssh || systemctl restart sshd

echo "SSH key injected for ${USER_NAME}"
```

然后再连：

```bash
ssh -i /home/shin/workspace/ssh-key-2026-05-06-2.key ubuntu@146.235.234.135
```

如果你的系统不是 Ubuntu，而是 Oracle Linux，用户名要改成：

```bash

```

也就是命令里的：

```bash
USER_NAME=opc
```

# 方案二：Serial Console 救援【中风险】

Oracle 官方支持 **Instance Console Connection**。

路径：

```text
Compute → Instances → Praxis → Console connection
```

或者：

```text
Resources → Console connection → Create console connection
```

这个不是普通 SSH，它是接入服务器的串口控制台。

如果你能进 Serial Console，恢复思路是：

1. 重启实例；
2. 在启动时进入 GRUB；
3. 进入 rescue/single user 模式；
4. 挂载根目录为可写；
5. 写入 `/home/ubuntu/.ssh/authorized_keys`；
6. 重启后正常 SSH。

核心命令是：

```bash
mount -o remount,rw /

mkdir -p /home/ubuntu/.ssh

cat >> /home/ubuntu/.ssh/authorized_keys <<'EOF'
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQC0Ijfmc3dB6CouEQ8P0pwQDk7us+RYUheoGf66WPOb7ag3fz9DDEl8rMZNH2R1bNcNKCyWbmI17U5KNSuEpEYIMImciFG52KrDvR6Z/9JxRWngXAzq6XehWqFdYA3Dbqh0qdZAylp8/+Qivy315++TyrOhRKoYo0NUZq1m6FvnA6oGJ977SaX0rzhl/OxeOx+QwbZ1i6DHELu8lXjsthoKl2i+PPpiap84ZWlYWZQjl2Zk5Thv5CtliHpFZP ssh-key-2026-05-06-2
EOF

chown -R ubuntu:ubuntu /home/ubuntu/.ssh
chmod 700 /home/ubuntu/.ssh
chmod 600 /home/ubuntu/.ssh/authorized_keys

reboot -f
```

但这个方案有个问题：Ubuntu 在 OCI 上有时不容易通过 Serial Console 进 GRUB。所以它不是最稳的。

# 方案三：Boot Volume 拆盘救援【最稳，但最后用】

这是 Oracle 官方博客推荐的兜底方案。

**重点：不是释放实例，不是删除实例。**

流程：

1. 停止 Praxis 实例；
2. 分离 Boot Volume；
3. 创建/使用另一台临时 Linux 实例；
4. 把 Praxis 的 Boot Volume 作为数据盘挂载到临时实例；
5. 修改里面的：

```text
/home/ubuntu/.ssh/authorized_keys
```

6. 卸载；
7. 把 Boot Volume 重新挂回 Praxis；
8. 启动 Praxis。

挂载后修复命令大概是：

```bash
sudo lsblk

sudo mkdir -p /mnt/praxis

# 具体分区可能是 /dev/sdb1 或 /dev/sdb3，需要 lsblk 判断
sudo mount /dev/sdb1 /mnt/praxis

sudo mkdir -p /mnt/praxis/home/ubuntu/.ssh

cat <<'EOF' | sudo tee -a /mnt/praxis/home/ubuntu/.ssh/authorized_keys
ssh-rsa AAAAB3NzaC1yc2EAAAADAQABAAABAQC0Ijfmc3dB6CouEQ8P0pwQDk7us+RYUheoGf66WPOb7ag3fz9DDEl8rMZNH2R1bThAZte+LSLbNsvjTm6f2xPR3WSvSs0w3aOSkqPkRxGcTscBQdI93TiiyvRBHTrSPc+Dj6FDZhBk7wkJKNCCSQ2VR1iNcNKCyWbmI17U5KNSuEpEYIMImciFG52KrDvR6Z/9JxRWngXAzq6XehWqFdYA3Dbqh0qdZAylp8/+Qivy315++TyrOhRKoYo0NUZq1m6FvnA6oGJ977SaX0rzhl/OxeOx+QwbZ1i6DHELu8lXjsthoKl2i+PPpiap84ZWlYWZQjl2Zk5Thv5CtliHpFZP ssh-key-2026-05-06-2
EOF

sudo chown -R 1000:1000 /mnt/praxis/home/ubuntu/.ssh
sudo chmod 700 /mnt/praxis/home/ubuntu/.ssh
sudo chmod 600 /mnt/praxis/home/ubuntu/.ssh/authorized_keys

sudo umount /mnt/praxis
```

这个方式几乎一定能恢复 SSH。

但风险是：A1 ARM 免费实例如果停机后再启动，极端情况下可能遇到容量不足。不是释放，但也有一点风险。所以我不建议第一步就用它。

# 第 9 步：顺手修 SSH 配置

```bash

PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
PasswordAuthentication no
KbdInteractiveAuthentication no
UsePAM yes
PermitRootLogin prohibit-password
EOF
```

```bash
if [ -f /mnt/praxis/etc/ufw/ufw.conf ]; then
```

## 设计方案：三层过滤

```
原始 session → 过滤器 → 有意义的 session → Obsidian vault（可检索）
                  ↓
           全量原始备份 → 冷存储（按日期归档，不进 git）
```

### Layer 1：过滤器（自动判断"有意义"）

判断标准——**有意义的 session 至少满足一条**：
- 用户发了消息（不是纯心跳）
- AI 返回了非空内容
- 包含工具调用（terminal/web/search 等）
- 包含文件操作、决策、错误记录
- 用户明确标记了 `archive=true`

**无意义 session 特征**：
- 纯心跳（只有 `heartbeat` 字样，无用户消息）
- 空内容（0 tool calls，AI 回复 < 50 字符）
- 连续重复的 "OK" 确认

### Layer 2：Obsidian Vault（只存有意义）

- 只进**过滤后**的 session
- 文件名包含 session_id + 日期 + 标签（如 `session_abc123_2026-05-07_有内容.md`）
- AI 需要回忆时，通过 **session_id 或关键词** 检索
- 总文件数预计降到 1,000-3,000 个

### Layer 3：全量原始备份（不占 git）

- 所有原始 session 压缩归档，按月份打包
- 存到 `/mnt/hindsight-sessions/YYYY-MM.tar.zst`
- 只在**需要排查问题时**解压查看
- 不进 Obsidian git，不影响同步
