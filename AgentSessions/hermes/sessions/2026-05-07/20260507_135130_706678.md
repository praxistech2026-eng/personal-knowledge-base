## YAML 全面审查报告

### 🔴 严重问题（会导致错误行为）

#### 1. 循环依赖 — `dialer-proxy` 自指
```yaml
- name: Webshare-美国
  dialer-proxy: 美国-链式前置   # ← 美国-链式前置 就是这个节点本身！
```
Webshare-US 的 `dialer-proxy` 指向 `美国-链式前置`，但 `美国-链式前置` 就是通过 filter 选中 Webshare-US 的那个组。**死循环**。

#### 2. 默认模板里的 RULE-SET 语法放错位置
```yaml
default: &default
  type: select
  proxies:
    - 直连
    - 所有-自动
    ...
    - RULE-SET,ChatGPT / Domain,AI服务   # ← 放在这里完全无效！
```
`RULE-SET,xxx` 是**规则语法**，不是 proxy group 语法。规则只能写在 `rules:` 段落，不能写在 `proxies:` 列表里。这个模板里的 `RULE-SET,ChatGPT...` 根本不会生效，而且会被 Clash 解析为"名为 RULE-SET,ChatGPT 的代理"——必定报错。

#### 3. `rule-anchor` 的 `type` 写法错误
```yaml
ip: &ip {type: http, interval: 86400, behavior: ipcidr, format: mrs}
   # ↑ type 应该是 "http" 字符串，不是对象 {type: http}
```
YAML anchor 引用的是**值**，不是 key。`<<: *ip` 展开后 `type` 的值变成嵌套对象 `{"type": "http", ...}`，这是错的。正确写法：
```yaml
ip: &ip
  type: http
  interval: 86400
  behavior: ipcidr
  format: mrs
```

### 🟡 功能性问题

#### 4. 所有策略组合并太粗糙
当前结构：
```
AI服务 → default模板 → 所有-自动/所有-手动/香港-故转/台湾-故转...
海外流媒体 → 同上
```

**问题**：AI 服务（Claude/ChatGPT）对 IP 纯净度**极度敏感**，需要单独节点池（Webshare 类），不应该和流媒体混用一个"所有-自动"。游戏同理，UBI 服务器在台湾，Steam 在美国，需要分开。

#### 5. TikTok 规则不够精细
TikTok 有地域限制（印度/中国禁售），只按域名分流不够。需要：
- `TikTok` → 策略组内要区分区域（比如"美国-故转"）
- 单独建 `TikTok非中` 策略组

#### 6. `lazy: false` 的意义理解错误
当前所有 fallback 组都用 `lazy: false`，意思是"即使节点可用也持续测速切换"。**这会额外消耗性能**。Fallback 设计是"备用"，应该 `lazy: true`（只有主节点挂了才切换），才符合"默认手动+自动回退"的核心逻辑。

#### 7. `class` 格式混用
`behavior: classical` 的规则集格式应该是 `text`，但引用了大量 `blackmatrix7/.../rule/Clash/XXX/XXX.list`。这些 `.list` 文件本身格式是 `domain:xxx` 或 `ip:xxx`，不是 mrs。混用 `class` + `text` 格式是对的，但要注意：
- `liandu2024/clash` 的 list 基本是 domain-only
- `blackmatrix7` 的规则是 domain + 少量 ip，behavior 应为 `domain`

#### 8. `rule-anchor` 命名冲突
```yaml
ip: &ip {...}
```
`&ip` 这个 anchor 名和 YAML 的 `ip:` key 重名，展开时可能覆盖 `ip` 字段。建议改为 `ip_anchor`。
