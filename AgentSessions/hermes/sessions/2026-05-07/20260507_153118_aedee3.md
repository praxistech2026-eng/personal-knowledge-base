## YAML 全面审查报告

### 🔴 严重问题（会导致错误行为）

#### 1. 循环依赖 — `dialer-proxy` 自指
```yaml
- name: Webshare-美国
  dialer-proxy: 美国-链式前置   # ← 美国-链式前置 就是这个节点本身！
```
Webshare-US 的 `dialer-proxy` 指向 `美国-链式前置`，但 `美国-链式前置` 就是通过 filter 选中 Webshare-US 的那个组。**死循环**。

#### 2. 默认模板里的 RULE-SET 语法放错位置
```yaml
default: &default
  type: select
  proxies:
    - 直连
    - 所有-自动
    ...
    - RULE-SET,ChatGPT / Domain,AI服务   # ← 放在这里完全无效！
```
`RULE-SET,xxx` 是**规则语法**，不是 proxy group 语法。规则只能写在 `rules:` 段落，不能写在 `proxies:` 列表里。这个模板里的 `RULE-SET,ChatGPT...` 根本不会生效，而且会被 Clash 解析为"名为 RULE-SET,ChatGPT 的代理"——必定报错。

#### 3. `rule-anchor` 的 `type` 写法错误
```yaml
ip: &ip {type: http, interval: 86400, behavior: ipcidr, format: mrs}
   # ↑ type 应该是 "http" 字符串，不是对象 {type: http}
```
YAML anchor 引用的是**值**，不是 key。`<<: *ip` 展开后 `type` 的值变成嵌套对象 `{"type": "http", ...}`，这是错的。正确写法：
```yaml
ip: &ip
  type: http
  interval: 86400
  behavior: ipcidr
  format: mrs
```

### 🟡 功能性问题

#### 4. 所有策略组合并太粗糙
当前结构：
```
AI服务 → default模板 → 所有-自动/所有-手动/香港-故转/台湾-故转...
海外流媒体 → 同上
海外游戏 → 同上
```

**问题**：AI 服务（Claude/ChatGPT）对 IP 纯净度**极度敏感**，需要单独节点池（Webshare 类），不应该和流媒体混用一个"所有-自动"。游戏同理，UBI 服务器在台湾，Steam 在美国，需要分开。

#### 5. TikTok 规则不够精细
TikTok 有地域限制（印度/中国禁售），只按域名分流不够。需要：
- `TikTok` → 策略组内要区分区域（比如"美国-故转"）
- 单独建 `TikTok非中` 策略组

#### 6. `lazy: false` 的意义理解错误
当前所有 fallback 组都用 `lazy: false`，意思是"即使节点可用也持续测速切换"。**这会额外消耗性能**。Fallback 设计是"备用"，应该 `lazy: true`（只有主节点挂了才切换），才符合"默认手动+自动回退"的核心逻辑。

---

### 🟢 需要补充的信源问题

#### 7. `class` 格式混用
`behavior: classical` 的规则集格式应该是 `text`，但引用了大量 `blackmatrix7/.../rule/Clash/XXX/XXX.list`。这些 `.list` 文件本身格式是 `domain:xxx` 或 `ip:xxx`，不是 mrs。混用 `class` + `text` 格式是对的，但要注意：
- `liandu2024/clash` 的 list 基本是 domain-only
- `blackmatrix7` 的规则是 domain + 少量 ip，behavior 应为 `domain`

#### 8. `rule-anchor` 命名冲突
```yaml
ip: &ip {...}
```
`&ip` 这个 anchor 名和 YAML 的 `ip:` key 重名，展开时可能覆盖 `ip` 字段。建议改为 `ip_anchor`。

基于你的实际场景：

|------|---------|---------|
| AI 服务（ChatGPT/Claude/Gemini）| IP纯净、低延迟、不频繁换IP | 单独节点池 + **手动优先** + fallback |

## 2. `Expire:2026-09-16` 为什么会出现在其他？

因为 `其他-手动 / 其他-自动` 的逻辑是：

> 只要不是美国、日本、香港、台湾、新加坡、韩国、英国、欧洲，就归到“其他”。

`Expire:2026-09-16` 不属于任何国家，所以被“其他”收进去。

v4 已修：`其他` 组也加了伪节点排除。

---

## 3. 为什么自动还能选择这种节点？按理说它应该不可选吧？

OpenClash/Mihomo 不知道它是不是“真节点”，它只看订阅 provider 解析出来的条目。

如果机场把下面这种东西也放进订阅：

```text
Traffic: 0.26 GB |150GB
Expire:2026-09-16
```

Mihomo 可能会把它当成一个 proxy entry。  
只要它进入了 provider，再被你的 `filter` 匹配到，`url-test` 就会把它列入自动组。

所以正确做法不是指望 Clash 自动识别，而是在配置里主动排除。

v4 已经做了这个排除。

## 4. `GLOBAL` 这个组是干什么的？

`GLOBAL` 通常不是我这个 YAML 里定义的组，而是 Clash/OpenClash 面板里的全局模式入口。

它的作用是：

> 当你把模式切成 Global / 全局模式时，所有流量都走 GLOBAL 当前选择的节点/策略。

你现在用的是：

```yaml
mode: rule
```

所以正常情况下主要走 `rules:` 分流，`GLOBAL` 不应该是日常关注重点。

简单说：

| 组 | 作用 |
|---|---|
| `GLOBAL` | 全局模式用，所有流量强制走一个出口 |
| `rule` 模式 | 按规则分流，才是我们现在这个 YAML 的核心 |

你日常不要动 `GLOBAL`，除非你想临时让所有流量都走同一个节点。

## 5. `国外` 这个组是干什么用的？

`国外` 是通用国外流量的兜底策略组。

比如这些规则会走它：

```yaml
RULE-SET,GFW,🌐 国外
RULE-SET,Proxy,🌐 国外
RULE-SET,Reddit,🌐 国外
RULE-SET,Wikipedia,🌐 国外
```

也就是说：

> 如果某个国外网站没有被更细的场景规则命中，就交给 `国外`。

它里面放的是普通机场节点，不放 Webshare 落地：

```yaml
🌐 所有-手动
⚡ 所有-自动
🇯🇵 日本-故转
🇭🇰 香港-故转
🇸🇬 新加坡-故转
🇺🇸 美国-故转
...
```

作用是避免所有东西都走 AI / 支付 / 视频 / 游戏这些专用组。

## 6. `国内` 这个组是干什么用的？

`国内` 是国内流量策略组。

规则里这些会走国内：

```yaml
China-Domain
China-IP
Private
Apple-CN
Direct
局域网 IP
```

默认：

但日常建议：

```text
国内 = 直连
```

## 7. `漏网之鱼` 这个组是干什么用的？

`漏网之鱼` 是最后的 `MATCH`：

```yaml
- MATCH,🐟 漏网之鱼
```

意思是：

> 前面所有规则都没匹配上的流量，最后全部进这个组。

它是兜底组。

比如某个新网站：

- 不在 China 规则
- 不在 GFW 规则
- 不在 AI 规则
- 不在视频/游戏/支付规则

最后就会进 `漏网之鱼`。

现在默认是：

```yaml
🐟 漏网之鱼:
  - 🌐 国外
  - 直连
  - 🌐 所有-手动
  - ⚡ 所有-自动
```

日常建议保持：

```text
漏网之鱼 = 国外
```

这样陌生国外网站不会漏直连。
