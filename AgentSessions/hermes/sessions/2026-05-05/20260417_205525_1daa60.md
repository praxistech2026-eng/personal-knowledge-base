### [] assistant

1. 总体架构

分三层，单向主链路 + 受控升级回路。

A. 执行层：OpenClaw
职责：
- 执行具体动作：读写文件、调用工具、运行命令、抓取页面、提交表单、调用 API
- 回传结构化结果，不做策略决策
- 每一步必须输出：
  1) action_id
  2) 输入参数
  3) 结果状态：success / fail / blocked
  4) 证据：stdout / 截图 / DOM / 返回值 / diff
  5) 可重试标记：retryable / non_retryable

边界：
- 不自主改目标
- 不自主扩任务
- 不自行进入长链推理
- 不写 memory

B. 决策层：Hermes
职责：
- 接收目标与约束
- 拆解任务为可执行步骤
- 给 OpenClaw 下发原子动作
- 校验执行结果是否满足预期
- 决定：继续执行 / 重试 / 改计划 / 升级 GPT / 终止
- 管理 memory 的写入和读取

核心节点：
- Planner：生成步骤序列
- Validator：校验 OpenClaw 返回证据是否满足验收条件
- Router：判断下一跳是 OpenClaw、GPT 还是 Terminate
- Memory Gate：只在满足写入条件时沉淀 memory

C. 关键推理层：GPT
职责仅限关键节点，不参与常规执行。
只处理三类输入：
- Hermes 无法在既定规则内完成的冲突判断
- 多方案高代价选择
- 需要高质量语义生成/重写/抽象压缩的节点

输出要求：
- 必须是受约束结构化结果：
  1) decision
  2) rationale_short
  3) confidence
  4) next_action
  5) retry_budget_update（如需）

主流程：
User Goal
-> Hermes Planner
-> OpenClaw Execute
-> Hermes Validator
-> 分支判断
   - 满足验收 -> 完成
   - 可修复且未超预算 -> Hermes 调整后继续派发 OpenClaw
   - 命中升级条件 -> GPT 输出决策 -> Hermes 执行决策
   - 命中终止条件 -> 终止

2. 升级到 GPT 的触发条件

只有满足以下精确条件之一才升级：

A. 计划冲突触发
- Hermes 生成的候选方案 >= 2
- 且每个方案在以下至少两个维度互相冲突：成功率、成本、时延、风险
- 且 Hermes 不能用预设优先级直接裁决
=> 升级 GPT 做方案选择

B. 重试失效触发
- 同一 action_type 在相同目标对象上连续失败 2 次
- 且两次失败的根因标签不同
- 且 Hermes 无法将问题归入已有修复模板
=> 升级 GPT 做根因归纳与下一步建议

C. 语义歧义触发
- 用户指令存在 >= 2 种可执行解释
- 且不同解释会导致不同外部副作用
- 且无法通过本地状态或历史 memory 消除歧义
=> 升级 GPT 生成最小歧义解释或最小澄清问题

D. 高价值输出生成触发
- 当前节点输出将直接面向用户、客户、审批流、正式文档或代码主分支
- 且该输出长度 > 200 字或涉及结构化论证/对外措辞
=> 升级 GPT 生成或润色最终文本/决策说明

E. 压缩触发
- 上下文材料 > 8 个独立证据块，或总长度 > 6KB
- Hermes 需要将其压缩成后续可用的决策摘要
=> 升级 GPT 做信息压缩

F. 风险触发
- 计划执行的下一步具备以下任一属性：
  1) 不可逆写操作
  2) 对生产环境生效
  3) 涉及权限提升
  4) 涉及资金、合规、法律声明
- 且 Hermes 无法用现有规则完成风险判定
=> 升级 GPT 做风险审查

明确不升级 GPT 的情况：
- 纯执行失败但根因明确
- 格式转换
- 常规重试
- 信息抓取
- 简单 if/else 路由
- 可由固定模板处理的报错

3. 哪些内容允许沉淀进 memory

只允许写入可复用、低争议、低时效敏感的信息，且必须附带来源与时间戳。

允许写入的具体类型：

A. 用户稳定偏好
- 输出语言偏好
- 输出格式偏好（表格/要点/JSON）
- 代码风格偏好
- 工具使用禁忌或优先级
- 审批阈值偏好（例如“涉及生产写操作先确认”）

B. 环境与资源映射
- 常用仓库路径
- 常用服务名称与对应环境
- 固定 API endpoint 别名
- 常用文件位置
- 固定账号角色映射（只存标识，不存密钥）

C. 已验证成功的操作模板
- 某类任务的稳定执行步骤
- 已证实有效的命令组合
- 已证实有效的页面操作序列
- 某系统的成功登录/提交流程特征

D. 已确认的业务规则
- 审批顺序
- 发布前置条件
- 命名规则
- 分支策略
- 文档归档规则
- 工单流转规则

E. 已确认的失败模式与修复映射
- 错误码 -> 修复动作
- 特定页面异常 -> 绕行步骤
- 特定接口返回模式 -> 处理办法

F. 长期任务状态锚点
- 当前项目阶段
- 已完成里程碑
- 待继续的明确下一步
- 被用户明确确认过的决策结果

禁止写入 memory 的内容：
- 临时 token、密码、cookie、密钥
- 一次性验证码
- 高时效状态（如“当前服务 CPU 90%”）
- 未验证的猜测
- 含歧义的用户意图
- 可从系统实时查询得到且变化频繁的数据
- 冗长原始日志、整段网页、完整对话逐字稿

4. 哪些情况必须终止而不是继续重试

以下场景必须立即终止，不继续重试：

A. 权限或身份边界明确不允许
- 返回 401/403 且当前账号身份已确认正确
- 页面或 API 明确提示无权限
- 需要更高角色/额外审批才能继续

B. 不可逆高风险操作缺少授权
- 将删除、覆盖、发布、付款、发信、提审、对外发送
- 且缺少用户明确确认或流程授权
- 即使技术上可执行，也必须终止

C. 目标对象不再存在或关键前提失效
- 目标文件/记录/页面/工单已删除
- 目标版本已变化导致原计划对象失配
- 必需依赖服务已下线且无替代路径

D. 输入数据已被证实自相矛盾
- 同一关键字段存在冲突且无法从系统判定真值
- 继续执行会导致错误写入或错误决策

E. 达到硬性重试上限
- 同一 action 在同一对象上连续失败 3 次
- 且至少 1 次已更换参数或策略
- 仍未成功
=> 终止，不再循环

F. 证据完整但结果不可修复
- OpenClaw 返回明确 non_retryable
- 如 schema 不兼容、字段已废弃、接口已下线、文件只读且不可变更

G. 外部系统给出强约束拒绝
- rate limit 明确要求冷却窗口且窗口超出当前任务 SLA
- 风控锁定
- 账号冻结
- CAPTCHA/人工验证拦截且当前流程无人工接入节点

H. Memory 与实时证据冲突且无法裁决
- 历史 memory 指向 A
- 当前系统证据指向 B
- Hermes 无法用本地规则判定哪个有效
=> 终止并上抛，不继续试错

I. 成本或时间预算耗尽
- 已达到预设 token / API 调用 / 执行时长上限
- 再尝试将违反任务预算或 SLA

J. 安全异常
- 检测到潜在数据泄露
- 意外命中生产环境
- 返回内容包含敏感凭据
- 执行动作与原目标发生明显偏移
=> 立即终止并冻结后续动作
