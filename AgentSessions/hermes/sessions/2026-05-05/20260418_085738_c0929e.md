我发现了什么

一、Obsidian 作为 PKM / 第二大脑的核心理念
1) Local-first + plain Markdown
- Obsidian 的基本单位是本地 Markdown 文件，数据归用户所有，不锁定在专有数据库里。
- 这对 AI/agent 非常重要：文件系统友好、易索引、易版本管理、易迁移。

2) Links over folders
- 官方设计重心不是“复杂层级目录”，而是通过 wikilinks、backlinks、graph 把知识组织成网络。
- 文件夹有用，但更多用于“边界/工作流分区”；知识结构本身应更多体现在链接与索引页上。

3) Emergent structure
- 不要求先设计完美分类法；允许先捕获，再逐步链接、提炼、聚合。
- 很适合 AI 辅助整理：先让 agent 做初步结构化，再由人复核。

4) Extensible workspace
- Obsidian 的核心能力很克制，复杂工作流通过插件实现。
- 这意味着它很适合做“AI orchestration layer”：模板、查询、自动化、知识提取都可外挂。

二、常见架构总结

1) 文件夹架构
常见做法：
- 尽量浅层：Inbox / Daily / Projects / Areas / Resources / Archive
- 或按笔记类型分：People / Concepts / Sources / Meetings / Tasks
- 避免过深层级；深层分类往往会和链接网络冲突

适合 AI 的建议：
- 让文件夹承担“生命周期/流程”职责，不承担全部知识语义
- 例如：
  - 00 Inbox
  - 10 Daily
  - 20 Projects
  - 30 Areas
  - 40 Resources
  - 50 Evergreen
  - 90 Archive

2) Wikilinks / Backlinks / MOC
- [[Note Name]] 是 Obsidian 知识网络核心。
- 最常见成熟实践是：原子笔记 + 索引页/MOC（Map of Content）
- MOC 不是“数据库”，而是人工维护的导航页/主题页。
- 对 AI 很关键：MOC 可以作为检索入口与上下文压缩层。

3) Dataview
- Dataview 本质上把 Markdown + frontmatter 变成“可查询知识库”。
- 适合汇总任务、项目状态、文献清单、实体列表、最近更新等。
- 它更像“read model / query layer”，适合 AI 读取、汇总、发现结构，不建议让 agent 过度依赖复杂动态查询结果写回原文。

4) Sync
- 官方 Obsidian Sync 强调端到端加密、多设备同步、版本历史。
- 对 AI 场景要重点考虑：
  - 哪些内容同步到云
  - 哪些内容允许发给外部 LLM
  - 是否采用本地 embedding / 本地索引 + 选择性上传

5) 插件生态
高频相关插件：
- Dataview：结构化查询
- Templater：模板与脚本
- QuickAdd：快速捕获/自动化
- Tasks：任务语义增强
- Calendar / Periodic Notes：时间维度组织
- Excalidraw / Canvas：可视化整理
- Omnisearch / Text Generator / AI 插件：增强搜索与生成
- Git：版本追踪与回滚

结论：
- Obsidian 的强项不是“单一最佳架构”，而是“Markdown 文件 + 链接网络 + 可选结构层 + 插件自动化”的组合。

三、适合与 AI agent / LLM Wiki 结合的最佳实践

1) 保持 vault 机器可读
推荐：
- 一条笔记一个主题/实体
- 稳定文件名或 note id
- 统一 frontmatter
- 固定字段，例如：
  title:
  type:
  status:
  tags:
  aliases:
  source:
  created:
  updated:
  related:
- 对 AI 来说，frontmatter 比纯自然语言更容易稳定解析

2) 把“链接”做成主索引，而不是只靠文件夹
- 让 agent 优先维护：
  - related links
  - MOC 页面
  - alias
  - 人物/项目/主题之间的关系
- 这样比自动塞进某个目录更有长期价值

3) 把 Dataview 当作“查询层”，不要当作唯一真相
- 真正的知识应存于 note/frontmatter 中
- Dataview 只负责展示、聚合、盘点
- AI 写入时应改源 note，不要只改查询页

4) 设计“人审阅”的 AI 工作流
推荐流水线：
- Capture：快速收集
- Normalize：agent 规范文件名、frontmatter、标签
- Link：agent 推荐 wikilinks / MOC
- Distill：agent 生成摘要、关键点、行动项
- Review：人确认并提升为 evergreen notes
- Publish/Query：Dataview/LLM Wiki 展示

5) 明确标记 AI 生成内容
建议增加字段：
- provenance: human | ai-assisted | ai-generated
- confidence:
- reviewed:
- source_url:
- extracted_from:
这样便于后续审校，也便于 LLM Wiki 区分“原始资料”和“AI 总结”。

6) 用模板约束 agent 输出
推荐对以下笔记类型做模板：
- 文献笔记
- 会议纪要
- 项目页
- 人物页
- 概念页
- 决策记录
模板越稳定，agent 越容易可靠写入。

7) 用 MOC / Hub pages 作为 AI 检索入口
- 不要让 agent 每次直接扫描整个 vault
- 先从项目页、主题页、MOC、近期日记入口做分层检索
- 这能降低上下文噪声，很适合 Hermes Agent + LLM Wiki

8) 安全与隐私分层
建议把 vault 分三层：
- Private：绝不送外部模型
- Assistive：可做摘要/分类
- Publishable：可进入 LLM Wiki 或公开知识库
- 若涉及敏感信息，优先本地索引/本地模型/脱敏后上传

9) 引入版本控制
- AI 自动改写时，强烈建议 Git 或 Obsidian Sync version history
- 这样可以回滚错误链接、错误 frontmatter、错误摘要

10) 不要让 agent 直接“重构整个 vault”
- 最好从小范围开始：
  - 先整理一个项目文件夹
  - 或先维护一个主题 MOC
  - 或先给 100 篇旧笔记补 frontmatter
- 全库自动重命名、批量移动目录风险很高

四、推荐给用户的落地架构（适合 Obsidian + Hermes Agent + LLM Wiki）
建议采用“轻文件夹 + 强链接 + 结构化 frontmatter + Dataview 查询”的混合架构。

推荐最小结构：
- 00 Inbox
- 10 Daily
- 20 Projects
- 30 Areas
- 40 Sources
- 50 Concepts
- 60 People
- 70 MOCs
- 90 Archive

每篇笔记尽量有：
- title
- type
- tags
- aliases
- status
- source
- related
- reviewed

Agent 职责建议：
- Hermes Agent：
  - 清洗新笔记
  - 生成摘要/关键词
  - 推荐 wikilinks
  - 维护项目页/MOC
  - 补 frontmatter
- LLM Wiki：
  - 作为“只读聚合层/问答层”
  - 从 MOC、Concept、People、Projects 中构建检索与问答
  - 不直接作为唯一写入源

简要结论
- Obsidian 最适合被理解为：本地 Markdown 知识文件系统 + 双链网络 + 插件化查询/自动化平台。
- 对 AI agent 最优的不是“复杂层级目录”，而是：
  1) 稳定的 Markdown/frontmatter
  2) 清晰的 note types
  3) 丰富 wikilinks / MOC
  4) Dataview 作为查询层
  5) Git/Sync 作为安全网
- 对 Hermes Agent + LLM Wiki 的最佳组合是：
  - Hermes 负责整理、提炼、链接、结构化写回
  - LLM Wiki 负责基于结构化 vault 做检索、问答、导航
  - 人负责最终审阅与知识升华
