1. 总体架构

1.1 分层定义

1) OpenClaw 执行层
- 定位：默认执行者、工具操作者、环境交互者。
- 负责任务类型：
  a. 工具调用类：终端执行、文件读写、脚本运行、Web 自动化、API 调用。
  b. 结构化流程类：按既定 SOP 执行多步骤任务。
  c. 可验证任务：结果能通过测试、状态码、日志、文件 diff、页面元素、返回值进行机器校验。
  d. 低风险局部决策：参数补全、重试、回滚、选择已知技能模板。
- 不负责：
  a. 高不确定性策略制定。
  b. 高风险最终判断。
  c. 缺少明确判定标准时的价值取舍。

2) Hermes 决策层
- 定位：守门人、编排器、路由器、风险控制器。
- 负责任务类型：
  a. 任务拆解：把用户目标拆成可执行子任务。
  b. 路由判断：决定交给 OpenClaw 直接执行，还是升级到 GPT。
  c. 风险评估：判断任务风险、可逆性、置信度、成本。
  d. 状态管理：跟踪 task_id、step_id、重试次数、失败类型、依赖关系。
  e. Memory 管理：决定什么可以沉淀为长期模式，什么必须丢弃。
- 不负责：
  a. 大量底层工具操作。
  b. 大规模原子执行。
  c. 在低价值场景中频繁调用 GPT。

3) GPT 关键节点介入层
- 定位：高价专家资源，只处理 Hermes 明确判定“低成本层无法可靠解决”的关键节点。
- 负责任务类型：
  a. 高复杂度推理：多约束冲突、目标不完整、方案权衡。
  b. 高风险判断：法律、合规、安全、财务、删除/迁移/发布等不可逆操作前的最终方案建议。
  c. 创意与抽象：架构生成、复杂文案、非模板化设计。
  d. 歧义消解：多个可行方案差异接近，而错误代价高。
- 不负责：
  a. 常规执行。
  b. 机械重试。
  c. 已能被规则系统或 OpenClaw 直接验证的普通问题。

  d. plan skeleton

2) Hermes 到 OpenClaw

- 传递内容：

  b. subtask
  c. acceptance_criteria
  d. tool_scope
  e. retry_budget
  f. data_access_scope

3) OpenClaw 到 Hermes
- 返回结构化执行结果：

  b. outputs
  c. artifacts

  e. error_code
  f. retry_count_used

- 返回结构化建议，不直接落地执行。
- Hermes 二次审查后，转换成可执行指令再发给 OpenClaw。

1.5 通信协议和数据边界

1) 协议要求
- 所有层之间统一使用 JSON 信封。
- 每次调用必须携带 task_id 和 step_id。
- 每个结果都必须带 status、error.code、evidence。
- 不允许返回“自由文本结论”作为唯一输出，必须有结构化字段。

2) 数据边界
- OpenClaw 只看执行上下文，不看全量长期记忆。
- GPT 只看 Hermes 过滤后的摘要，不直接访问原始环境、密钥、完整用户数据。
- 只有 Hermes 能读写长期 memory。
- 原始日志、完整 trace、敏感输入保留在短期运行上下文，不进入长期层。

3) 权限边界
- OpenClaw：执行权限。
- Hermes：决策权限、路由权限、memory 写权限。
- GPT：建议权限，不拥有执行权限和记忆写入权限。

2. GPT 升级触发条件（精确的）

以下条件均为“当满足条件 X 或 Y 时，升级到 GPT”。

1) 置信度触发
- 当 Hermes 对当前路由或判断的 confidence_score < 0.70 时，升级到 GPT。
- 当 OpenClaw 返回 confidence_local < 0.60 且结果无法通过外部验证时，升级到 GPT。

2) 重试失败触发
- 当同一 step_id 在相同参数族下连续重试 >= 3 次仍未达到 acceptance_criteria 时，升级到 GPT。
- 当同一 task_id 在 10 分钟窗口内出现 retryable_fail >= 5 次时，升级到 GPT。

3) 错误码触发
- 当 error.code 命中以下集合之一时，升级到 GPT：
  a. AMBIGUOUS_GOAL
  b. CONFLICTING_CONSTRAINTS
  c. HIGH_IMPACT_DECISION
  d. SAFETY_BORDERLINE
  e. COMPLIANCE_UNCLEAR
- 当 error.code 命中 AUTH、PERMISSION、RATE_LIMIT、NETWORK_UNREACHABLE 时，不直接升级 GPT，优先走规则化处理；仅当规则处理失败次数 >= 2 时再升级。

4) 高风险任务类型触发
- 当 task_type 命中 high_risk_category 集合时，升级到 GPT。
- high_risk_category 的可编程集合建议定义为：
  a. delete_production_data
  b. financial_commitment
  c. legal_or_policy_interpretation
  d. security_rule_change
  e. user_facing_mass_send
  f. irreversible_migration
  g. medical_or_safety_advice_generation
- 命中即升级，不需要等待失败。

5) 不可逆操作前触发
- 当 planned_action.irreversible == true 且 expected_impact_scope >= 100 records 或 >= 1 production system 时，升级到 GPT。
- 当 rollback_plan == null 且 write_operation == true 时，升级到 GPT。

6) 方案竞争触发
- 当 Hermes 生成的候选方案数量 >= 2，且 top1_score - top2_score <= 0.10 时，升级到 GPT。
- 当多方案的预估风险差异 <= 0.05 且收益差异 <= 0.10 时，升级到 GPT。

7) 需求歧义触发
- 当需求解析后 mandatory_slots 缺失数 >= 2 时，升级到 GPT 或向用户澄清；如果任务属于高风险类，则优先升级 GPT。
- mandatory_slots 示例：目标对象、操作类型、约束边界、成功标准。

8) 创意/抽象度触发
- 当任务类型属于 architecture_design、brand_positioning、novel_content_creation、policy_synthesis 时，且不存在现成模板命中率 >= 0.80，则升级到 GPT。

9) 用户显式要求触发
- 当用户消息包含显式升级标记，如“用 GPT 评估”“调用更强模型”“让 Claude/GPT 来判断”时，升级到 GPT。
- 当用户要求“给出最终方案论证”且本地仅能给出执行结果不能给出高质量论证时，升级到 GPT。

10) 验证冲突触发
- 当至少两个独立验证器返回互相矛盾结论，且冲突无法通过规则消解时，升级到 GPT。
- 可编程判定：validator_agreement_ratio < 0.67。

11) 长上下文压缩损失触发
- 当输入上下文压缩后，关键约束保留率 estimated_retention < 0.85 时，升级到 GPT 进行高质量整合，或要求用户分批输入。
- estimated_retention 可由约束抽取器统计“原始约束数 vs 压缩后保留数”得到。

12) 成本阈值触发
- 当继续本地重试的预估成本 > 单次 GPT 调用成本的 1.5 倍，且成功率预测 < 0.40 时，升级到 GPT。
- 这是避免低价层盲目消耗的经济型触发器。

3. memory 沉淀规则

3.1 允许沉淀

1) 可复用偏好
- 例如：用户偏好输出语言、回答风格、交付格式、是否喜欢先结论后细节。
- 原因：稳定、低敏感、高复用，能持续提升后续交互效率。

2) 已验证工作模式
- 例如：某类任务的高成功率 SOP、某种错误的最佳恢复路径、某类仓库常见构建顺序。
- 原因：沉淀的是“模式”，不是单次原始数据，可降低未来成本。

3) 稳定环境事实的抽象
- 例如：用户常用工作目录模式、常见工具栈、长期项目结构标签。
- 原因：属于长期上下文，能帮助更快路由，但应以摘要形式保存。

4) 风险与边界偏好
- 例如：用户不希望自动修改生产配置、偏好先 dry-run、偏好非破坏性操作。
- 原因：这是安全策略，长期有效，直接影响决策质量。

5) 成功案例的抽象索引
- 例如：对“Python 项目依赖修复”建立 problem_pattern -> solution_pattern 映射。
- 原因：保留成功模式，不保留原始文件内容，便于未来检索复用。

6) 长期目标与角色信息
- 例如：用户正在做的长期项目类型、团队角色、主要语言栈。
- 原因：帮助 Hermes 更准确地规划和选择工具。

7) 用户明确授权持久化的信息
- 条件：用户显式说“记住这个偏好/规则/项目背景”。
- 原因：授权明确，适合进入长期记忆。

允许沉淀的数据形态要求：
- 必须是抽象后的事实、规则、偏好、模式、标签。
- 必须满足：跨会话复用概率 >= 0.5，敏感等级 <= low，且不依赖单次上下文才能成立。
- 建议结构：
{
  "memory_type": "preference|pattern|constraint|project_context",
  "subject": "...",
  "abstracted_fact": "...",
  "evidence_count": 3,
  "sensitivity": "low",
  "ttl": "long|medium",
  "source_scope": "multi_session|explicit_user_instruction"
}

3.2 禁止沉淀

1) 原始用户数据
- 例如：完整文件内容、原始日志、聊天原文、上传文档全文、网页抓取全文。
- 原因：高泄露风险，且长期复用价值低于风险。

2) 凭证与秘密
- 例如：API key、token、cookie、密码、私钥、session id。
- 原因：绝对敏感，禁止进入长期记忆。

3) 个人敏感信息
- 例如：身份证号、电话、邮箱、住址、支付信息、医疗信息、隐私偏好细节。
- 原因：隐私高风险，除非系统有专门合规机制，否则禁止沉淀。

4) 失败 trace 与完整报错栈
- 例如：stderr 全文、调用链、崩溃转储、请求体、响应体。
- 原因：常包含敏感内容，且噪声大；只允许抽象成“错误模式”。

5) token 消耗记录与底层推理痕迹
- 例如：prompt 全文、chain-of-thought、模型内部草稿、token 开销细目。
- 原因：不稳定、不可复用、合规风险高。

6) 一次性临时状态
- 例如：本次运行的临时路径、短期 session 缓存、瞬时 UI 状态、临时下载链接。
- 原因：过期快，长期价值低。

7) 未验证结论
- 例如：猜测性的用户偏好、单次观察形成的规则、没有证据支撑的环境推断。
- 原因：会污染长期记忆，导致未来错误决策。

8) 高风险业务数据
- 例如：客户名单、交易记录、数据库样本、内部策略文档原文。
- 原因：即使有帮助，也应只保留抽象标签，不能保存原件。

禁止沉淀的硬规则
- 当 sensitivity >= medium 时，默认禁止。
- 当内容包含 secret_pattern 命中时，直接禁止。
- 当内容无法被抽象成“去标识化模式”时，禁止。
- 当内容仅在单次任务有效，ttl <= session 时，禁止。
- 当内容来源于失败步骤且未被验证修正时，禁止。

4. 终止条件（非重试红线）

以下条件一旦触发，必须终止当前工作流，而不是继续重试。

1) 用户授权缺失
- 精确条件：action.requires_confirmation == true 且 user_confirmation != granted。
- 例子：删除、发布、覆盖、付费、对外发送、生产变更。

2) 明确安全/合规阻断
- 精确条件：error.code in {SAFETY_BLOCK, POLICY_BLOCK, COMPLIANCE_DENY, LEGAL_RESTRICTED}。
- 处理：直接停止，并返回阻断原因。

3) 权限不可获得
- 精确条件：error.code in {AUTH, PERMISSION} 且 credential_source == none 且 escalation_path == none。
- 含义：没有凭证、也没有获取路径，继续重试无意义。

4) 输入先验不足且不可补足
- 精确条件：mandatory_slots_missing >= 1 且 retrievable_by_tools == false 且 user_not_available == true。
- 含义：关键参数缺失，既无法查到，也无法问到。

5) 外部依赖永久性失败
- 精确条件：dependency_status in {DEPRECATED, REMOVED, HARD_404, ACCOUNT_CLOSED}。
- 含义：依赖对象已不存在，不应重试。

6) 数据边界被突破
- 精确条件：required_data_scope > granted_data_scope。
- 例子：任务要求访问未授权目录、额外系统、他人数据。
- 处理：停止，不自动扩大权限。

7) 不可逆操作且无回滚方案
- 精确条件：planned_action.irreversible == true 且 rollback_plan == null 且 user_confirmation != granted。
- 处理：直接终止当前流程，不进行试探执行。

8) 成本预算耗尽
- 精确条件：spent_budget >= max_budget。
- 预算可为：调用次数、时间、金钱、GPT 配额。
- 含义：继续执行违反系统资源约束。

9) 死循环判定
- 精确条件：最近 3 次尝试的 normalized_error_signature 完全相同，且输入参数变化率 < 10%。
- 含义：说明只是重复撞墙，不是有效搜索。

10) 验证器证明目标不成立
- 精确条件：formal_verifier_result == impossible 或 precondition_check == false 且该前置条件不可满足。
- 例子：请求的文件不存在且来源不可恢复；目标站点没有该入口；语法目标与运行环境版本不兼容且无法升级。

11) Memory 写入违规
- 精确条件：memory_candidate.sensitivity >= medium 或 memory_candidate.contains_raw_user_data == true。
- 含义：该流程中的记忆沉淀步骤必须停止，且不应通过“重试清洗”继续自动写入，除非进入人工确认/严格脱敏流程。

12) 用户明确要求停止
- 精确条件：收到 stop/cancel/不要继续/先到这里 等终止意图。
- 含义：立即停止当前工作流，不进行后续自动动作。

13) 高风险结论仍不确定
- 精确条件：task.risk_level == high 且 final_confidence < 0.75 且 GPT 已调用 >= 1 次仍无确定建议。
- 含义：已经使用高价专家仍不可靠，应该停止并返还人工决策。

14) 环境破坏风险上升
- 精确条件：连续两次失败后 side_effect_detected == true 且 side_effect_reversible == false。
- 含义：继续尝试可能进一步损坏环境。

5. 推荐的最小运行策略

1) 默认路径
- Hermes 先做任务分类、风险扫描、置信度估计。
- 满足“可执行且低风险”时，直接派给 OpenClaw。
- OpenClaw 执行并回传证据。
- Hermes 验证通过后输出结果，并仅抽象沉淀模式。

2) 升级路径
- 一旦命中精确升级条件，Hermes 只把“决策问题”升级给 GPT，而不是把整个执行过程都交出去。
- GPT 给出方案建议后，Hermes 把建议转译成可执行指令，再由 OpenClaw 落地。

3) 终止路径
- 一旦命中非重试红线，Hermes 立即停止，不再做等价重试。
- 返回终止原因、触发条件、缺失前提或需要的人类输入。

6. 设计总结

1) OpenClaw 最大化承担“能做、能验、能重试”的执行工作。
2) Hermes 最大化承担“低成本判断、风险守门、路由和记忆治理”。
3) GPT 被严格限定在“高不确定、高风险、高抽象价值”的关键节点上。
4) Memory 只沉淀抽象模式，不沉淀原始数据。
5) 终止条件必须硬编码，防止系统在高成本和高风险情境中死循环。
